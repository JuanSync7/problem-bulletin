# v2.12-WP03 (C1+C2) — ORM `to_dict()` ⊆ matching `*Read` schema contract pins

## Motivation

v2.11 retrospective cross-cutting lesson #2: "Schema convergence narrows
surfaces silently." WP06 added `Page[T]` and WP07 added `response_model=`
on adjacent routes; in between, `TicketRead` got narrowed without anyone
noticing. The OpenAPI introspection tests caught it within one WP, but
the right invariant is a contract test across every
`(ORM_class, schema_class)` pair. This WP delivers that test.

## Scope

- One collected pytest case per `(ORM, *Read)` pair, parametrized by id
  `"<ORM><->Schema>"` for legible failures.
- No conftest hooks. No production code changes (no latent narrowing
  bugs were found — the WP07 fix already exhausted the known surface).

## ORM-to-Schema mapping (audit output for G1)

Discovered eight `to_dict()` methods under `app/models/`:

| ORM class | File | Schema | Polarity | Pair status |
|-----------|------|--------|----------|-------------|
| `Ticket` | `app/models/ticket.py` | `TicketRead` | open (`extra="allow"`) | Pinned: required-fields-present |
| `Project` | `app/models/project.py` | `ProjectRead` | closed | Pinned: subset |
| `Sprint` | `app/models/project.py` | `SprintRead` | closed | Pinned: subset |
| `Component` | `app/models/project.py` | `ComponentRead` | closed | Pinned: subset |
| `ProjectMember` | `app/models/project.py` | `ProjectMemberRead` | closed | Pinned: subset |
| `TicketWatcher` | `app/models/ticket_watcher.py` | `TicketWatcherRead` | open (`extra="allow"`) | Pinned: required-fields-present |
| `TicketAttachment` | `app/models/ticket_attachment.py` | `TicketAttachmentRead` | open (`extra="allow"`) | Pinned: required-fields-present |
| `TicketNotification` | `app/models/ticket_notification.py` | `TicketNotificationRead` (see below) | n/a | **Unmatched** — not the same wire shape |

**Unmatched: `TicketNotification` ↔ `TicketNotificationRead`.**
The ORM emits `actor_id` + `actor_type` (raw FK pair). The Pydantic
schema replaces those two keys with a fully-hydrated `actor: PersonRef`
that the inbox route batch-resolves via `PeopleService` to avoid N+1
(see `app/schemas/notifications.py` module docstring). Routes never
serialize `TicketNotification.to_dict()` directly to the wire — they
project to `TicketNotificationRead` via service-layer hydration. A
naive `to_dict().keys() ⊆ model_fields.keys()` lint would produce a
false positive here (`actor_id`/`actor_type` would look like
"narrowed-away" keys). Documenting the pair as **intentionally
non-equivalent** and excluding from parametrization, per WP rule #1
("record those and continue; do not fabricate pairs").

A future WP wanting to lint the hydrated-pair contract would need a
different invariant (e.g. "every PersonRef field plus `kind`/
`recipient_*`/`target_*`/`is_read`/`created_at` is produced by the
hydration helper"). Out of scope here.

## Polarity decision (G2)

The v2.11-WP07 failure mode was: `Ticket.to_dict()` produced keys that
`TicketRead` silently dropped on the wire. So the closed-schema lint
is:

    set(instance.to_dict().keys()) <= set(SchemaCls.model_fields.keys())

For open schemas (`model_config = ConfigDict(extra="allow")`), the
subset constraint is vacuous (the schema accepts anything). The lint
flips polarity:

    set(required_fields(SchemaCls)) <= set(instance.to_dict().keys())

Both polarities together pin both narrowing directions: schema removing
a key, *or* ORM removing a key. This catches WP07's exact failure mode
on the four closed schemas and the analogous failure mode on the four
open ones.

## Instantiation strategy

`to_dict()` reads attributes only — no DB session needed. The test
instantiates each ORM via direct attribute assignment (`p = Project();
p.id = uuid4(); ...`) with every column the `to_dict()` body touches
populated by a sensible default. Pure-Python, fast (~0.33s for 7
collected tests).

The new WP02 `tests/helpers/source_lint.py` helper was considered and
**not used** — it is AST-oriented (source-shape lint), whereas this WP
is import-and-introspect (Pydantic model-field reflection + ORM
attribute access). Force-fitting would have invited more code than it
saved. Per WP rule #4.

## Latent narrowing bugs found

**None.** All 7 collected pairs pass first time.

- Closed-schema subset passes: every key produced by `Project.to_dict()`
  / `Sprint.to_dict()` / `Component.to_dict()` / `ProjectMember.to_dict()`
  is declared on the corresponding `*Read` schema.
- Open-schema required-fields-present passes: every required field on
  `TicketRead` / `TicketWatcherRead` / `TicketAttachmentRead` is
  produced by the corresponding `to_dict()`. (`TicketRead` notably
  got `extra="allow"` in v2.11-WP07 specifically because of the
  narrowing bug; this WP confirms that fix held.)

The contract is now pinned for future WPs — any narrowing of either
side fails at collection-time-adjacent test-time, not at frontend
regeneration time.

## Deferred / scope cut

- **`TicketNotification` hydrated-shape lint.** Different invariant
  (PersonRef hydration completeness), out of scope. Recorded above
  for future consideration if a third hydrated-projection schema
  surfaces — v2.11 cross-cutting lesson "no premature abstraction."
- **`ProjectMemberRead` has no factory-level invariant for
  `member_type='agent'` path.** The factory exercises the `'user'`
  branch only. If a future ORM `to_dict()` is gated on
  `member_type`, add a second factory variant. Not load-bearing
  today since `to_dict()` returns `self.member_type` verbatim.
