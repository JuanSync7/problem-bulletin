# v2.14-WP03 (B4) — Parity-lint performance pass

## Goal

Profile the OpenAPI↔TS parity-lint cluster
(`tests/test_openapi_ts_parity_*.py`) and, if cluster wall is >2s or any
individual test creeps multi-second, consolidate the boot/parse fixtures
into session-scoped fixtures.

Baseline after v2.14-WP02 just landed: backend **1429 P / 0 F / 5 skipped
/ 14 xfailed**, frontend **243 P / 0 F**.

---

## Pre-pass timings (baseline)

Command:

```
.venv/bin/pytest tests/test_openapi_ts_parity_*.py --durations=0 -v
```

Result: **27 passed in 2.01s**

`--durations=0` slowest (everything else <0.005s, hidden by pytest):

```
1.31s setup    tests/test_openapi_ts_parity_wp05_v213.py::
                test_page_wrapper_parity[GET /api/v1/projects:...]
0.44s setup    tests/test_openapi_ts_parity_wp11.py::
                test_openapi_ts_parity[GET /api/v1/projects/{id_or_key}:
                ProjectRead<->ProjectDTO]
```

Read: each parity-lint module had its own `@pytest.fixture(scope="module")
def app(): return build_test_app()` plus `def openapi_spec(app): return
app.openapi()`. Both files boot the FastAPI app independently, so the
session pays the boot+OpenAPI-schema build twice — once per file. Total
setup cost across the cluster ≈ 1.75s, dominating the 2.01s wall.

---

## Decision

**Consolidate.** Cluster wall 2.01s is at the seed's defer/consolidate
threshold (">2s cluster, or individual tests creep into multi-second
territory"), and individual setup costs (1.31s, 0.44s) clearly exceed
the 0.3s defer ceiling. The work is mechanically simple: the two module
fixtures already produce identical objects (same `build_test_app()`,
same `app.openapi()` schema), so a session-scoped fixture is a strict
de-duplication.

The seed cautioned against speculative consolidation. This is not
speculative: the duplicate boot is observed twice in the durations
profile and the savings are deterministic.

---

## What changed

1. Added three session-scoped fixtures to `tests/conftest.py`
   (additive only — no existing fixture modified):

   - `parity_lint_app` — single `build_test_app()` call per session.
   - `parity_lint_openapi_spec` — single `app.openapi()` call per session.
   - `parity_lint_ts_sources` — pre-read dict of frontend TS sources
     (`tickets.ts`, `projects.ts`, `sprints.ts`, `notifications.ts`,
     `search.ts`, `people.ts`, `users.ts`, `comments.ts`). Reserved
     for future use; not yet consumed by either parity-lint file (the
     existing `_read_ts` helpers are sub-millisecond — keeping the
     in-module helpers intact avoids touching test-level surface area
     that isn't on the perf-critical path).

2. Migrated `tests/test_openapi_ts_parity_wp11.py` and
   `tests/test_openapi_ts_parity_wp05_v213.py` to consume
   `parity_lint_app` / `parity_lint_openapi_spec` via thin
   module-scoped re-export fixtures:

   ```python
   @pytest.fixture(scope="module")
   def app(parity_lint_app) -> FastAPI:
       return parity_lint_app

   @pytest.fixture(scope="module")
   def openapi_spec(parity_lint_openapi_spec) -> dict[str, Any]:
       return parity_lint_openapi_spec
   ```

   This preserves every test's existing parameter signature (`app`,
   `openapi_spec`) — zero call-site changes. The module-level fixture
   becomes a no-op pass-through and the real cost is paid once per
   session.

3. Left `build_test_app` import in both files; it is no longer the
   active producer but kept to make the pattern's origin obvious to
   readers (intentionally minimal diff).

## Canonical consolidation recipe (for future WPs)

If a new parity-lint module is added (e.g. `test_openapi_ts_parity_wp<N>.py`)
and the per-module setup cost re-appears, follow this recipe:

1. **Profile**: `pytest tests/test_openapi_ts_parity_*.py --durations=0 -v`.
   Defer if cluster ≤1s OR no individual test/setup >0.3s.

2. **If consolidating**: do NOT add another `@pytest.fixture(scope="module")
   def app()` in the new file. Consume the session-scoped fixtures from
   `tests/conftest.py`:

   ```python
   # New parity-lint file
   @pytest.fixture(scope="module")
   def app(parity_lint_app) -> FastAPI:
       return parity_lint_app

   @pytest.fixture(scope="module")
   def openapi_spec(parity_lint_openapi_spec) -> dict[str, Any]:
       return parity_lint_openapi_spec
   ```

3. **For TS source reads >50ms per file**: prefer the
   `parity_lint_ts_sources` session-scoped dict (already in
   `tests/conftest.py`). Extend its file list if you add a new
   `frontend/src/api/*.ts` consumer. Below 50ms per file the in-module
   `_read_ts` helper is fine — disk reads are cheap, but the FastAPI
   boot dominates.

4. **Threshold for the next consolidation pass**: re-run perf if
   cluster wall creeps back above ~2.5s (giving room for one more
   parity-lint module's boot cost before it becomes painful).
