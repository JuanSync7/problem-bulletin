## Parser edge cases handled

1. **`any` in string literals / identifiers**: AST walk matches
   `SyntaxKind.AnyKeyword` nodes only. The literal string `'any'` (e.g.
   `PersonPicker`'s `kind: 'user' | 'agent' | 'any'`) and identifiers
   containing the substring `any` (`many`, `anything`) are not flagged.
   Self-test 8 pins this.
2. **`any` inside comments / JSDoc**: AST walk skips comments entirely.
3. **`as any` cast**: covered by AST walk because `AsExpression` carries
   an `AnyKeyword` TypeNode child. Self-test 2 pins this.
4. **Generic arg `any`**: `Array<any>`, `Record<string, any>`,
   `Foo<any>` all expose an `AnyKeyword` TypeNode under
   `TypeReference.typeArguments`. Self-test 3 pins this.
5. **Function-type / return-type `any`**: `(x: any) => ...` and `():
   any` both have `AnyKeyword` as the type slot. Self-test 1 pins the
   param case.
6. **Per-line dedupe**: if a line contains multiple `AnyKeyword` nodes
   (e.g. `(x: any): any =>`), the lint reports only one offender per
   line so allow-list entries are stable across small whitespace edits.
   This is implemented via a `seen: Set<line>` in `scanSource`.
7. **`@ts-*` directives**: compiler API does not surface these as AST
   nodes (they are comment-pragmas handled out-of-band). A bounded
   per-line regex `/[\/*]\s*@ts-(ignore|expect-error|nocheck)\b/`
   matches both `//` and `/*` comment styles. Self-tests 4-6 pin each
   directive kind.
