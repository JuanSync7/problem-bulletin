**Pattern read.** The dominant clusters are:

1. **SQLAlchemy `Column[...]` leaking past ORM boundary** (most
   `arg-type` / `attr-defined` hits) — mypy treats raw `Column[UUID]`
   as not assignable to `UUID`, and treats lazy-loaded list relationships
   as `list?[T]` which is not iterable. These are exactly the cases
   pyright resolves better via the SQLAlchemy 2.0 typed-Mapped plugin.
2. **`list?[T]` (Optional list) iterated unconditionally** — surfaces
   in `app/routes/tickets.py`, `app/routes/projects.py`. Real fixable
   bugs; either tool would catch.
3. **Pydantic v2 `Page[T]` mismatch** — `Page(items=[dict, ...])`
   vs `list[XRead]`. Real DTO drift; either tool catches.
4. **Missing stubs** — `python-jose` has no stubs (`import-untyped`).
   Cheap to silence with a per-module `ignore_missing_imports`.
5. **Self-shadowing**: `TicketService.list` (the method) shadowing
   `typing.List` style annotations in 4 spots — `[valid-type]` cluster
   in `app/services/tickets.py`. Easy targeted fix.

## Recommendation for WP02

**Mypy** — keep it. One-line rationale:

> Mypy is already installed and in `pyproject.toml`; the dominant
> error clusters (SQLAlchemy `Column[T]` leaks, Pydantic-v2 `Page[T]`
> mismatches) are not pyright-exclusive wins on this codebase, and the
> 170-error surface is small enough to gate in CI by writing a
> `[tool.mypy]` config block + a targeted allow-list (mirror of the
> v2.17-WP03 `# type: ignore` PIN) rather than adopting a second
> toolchain. Pyright would also work, but adopting it now costs an
> install + a parallel config + npm/node coupling on the backend — no
> proportional payoff visible in the sniff.

**WP02 scope sketch** (informs the WP02 plan, NOT executed here):

1. Add `[tool.mypy]` block in `pyproject.toml` with sensible defaults
   (`python_version = "3.12"`, `warn_unused_ignores = true`,
   `ignore_missing_imports = false` globally, per-module override for
   `jose` and any other untyped third-party).
2. Triage the 170 errors into BY-DESIGN / LEGACY split — pattern from
   the WP03 allow-list. Estimated split based on the sniff:
   - **Real bugs / easy fixes** (~30 errors): `list?[T]` iterations,
     `Page[T]` item-type mismatches, the `TicketService.list`
     self-shadow. Fix in place.
   - **SQLAlchemy `Column[T]` boundary** (~80 errors): wrap with a
     small `as_uuid()` / `as_str()` adapter, OR mark whole files
     with a per-module mypy override during the PIN; revisit when
     migrating to SQLAlchemy 2.0 Mapped annotations.
   - **Pydantic-v2 model construction edges** (~40 errors): targeted
     `cast()` or model-config tweaks.
   - **Stubs / untyped imports** (~10 errors): per-module
     `ignore_missing_imports`.
   - **Genuine LEGACY suppressions** (~10 errors): `type: ignore` +
     LEGACY allow-list entry, deferred to a later WP.
3. Wire `mypy app/` into the CI lint job (alongside ruff), gating
   on the snapshot allow-list — exactly mirroring v2.17-WP03's PIN
   shape so the pattern is consistent across linters.

Effort estimate: WP02 sized for one WP if we PIN-not-SWEEP (mypy as
CI gate against a frozen 170-error snapshot, fix the ~30 easy bugs in
the same WP, defer the SQLAlchemy 2.0 Mapped migration to a future
v2.20+ WP).
