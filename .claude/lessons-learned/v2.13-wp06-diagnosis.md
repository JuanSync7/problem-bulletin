# v2.13-WP06 (B1) — `entity=all` `refresh_total` semantics

## Decision

**Option (b): all-arms-or-none.** When `refresh_total=1` is passed
alongside `entity=all`, every arm gets a fresh `COUNT(*) OVER ()`
recount and surfaces `total_authority="live"`. There is no per-arm
opt-in syntax.

## Rationale

1. **UX-fit.** The `entity=all` tab is a 5-per-arm preview meant for
   orienting users — they typically pivot to a single-arm tab to
   drill in. A per-arm refresh would force the user to think about
   which arm to refresh; that's the wrong cognitive load for an
   overview tab. The single-arm tabs already carry the v2.12-WP10
   banner/button for fine-grained refresh.

2. **Drift is mostly invisible on `entity=all`.** Users see only 5
   items per arm; the totals are auxiliary. If a user cares about an
   arm's count enough to refresh, they're already going to switch to
   that arm's dedicated tab — where v2.12-WP10's button is wired.

3. **Cost/benefit.** Option (b) is ~5 LOC backend (already-correct;
   needed only verification + a route test) + ~10 LOC frontend
   (hook + page banner). Option (a) requires a new query-param
   shape (`refresh_total=problems,tickets`), a new union type
   (`refresh_total: boolean | string[]`), and per-arm UI affordances.
   The complexity multiplier is ~5x for marginal value.

4. **Forward door is open.** Each arm's `total_authority` already
   reflects whether THAT arm got a fresh count (it does today —
   `refresh_total` is threaded through to every per-arm service
   call). If a future second consumer surfaces a clear need for
   per-arm, the data is already on the wire — upgrade path is a
   query-param shape change, not a model change.

## Implementation summary

### Backend

- `app/services/search_multi.py::search_entities` **already**
  propagates `refresh_total` to every arm when `entity="all"`
  (lines 322, 335, 347, 359, 369 thread the same `refresh_total`
  kwarg into each `_search_<arm>` call). No code change required.
- `app/routes/search.py::search_v2` **already** accepts
  `refresh_total: bool = Query(False, ...)` and passes it through
  to `search_entities`. No code change required.
- **Test added.** `tests/routes/test_search_v2_refresh_total_wp14.py`
  gains one test (`test_refresh_total_all_recounts_every_arm`)
  that seeds rows across two arms (problems + users), pages once
  with `entity=all&refresh_total=1` after deleting rows, and
  asserts every present arm reports `total_authority="live"` AND
  the live (post-delete) totals. Filenames extend the WP14 test
  module (the natural home — same query param, complementary
  coverage).

### Frontend

- `frontend/src/hooks/useSearchV2.ts` — `totalAuthority` now also
  surfaces a value for `entity="all"`: returns `"snapshot"` when
  any present arm reports snapshot, `"live"` only when every
  present arm reports live. This collapses the multi-arm shape
  into the existing banner's binary trigger. `refreshTotal()`
  works without modification — the one-shot `refreshPendingRef`
  fires the next request with `refresh_total=true` regardless of
  entity, and the backend already broadcasts.
- `frontend/src/pages/Search.tsx` — banner condition extended.
  For `entity="all"`, banner shows whenever `totalAuthority ===
  "snapshot"` (no `hasPrev` predicate — there is no cursor chain).
  Label and button reuse the v2.12-WP10 strings (banner reads
  "Showing snapshot counts" plural variant on the All tab).
- The hook contract stays minimal — no new fields, no new
  abstractions. The "All tab" case is a derived value in the same
  `totalAuthority` field.

## What changed vs the v2.12-WP10 contract

| Aspect | v2.12-WP10 | v2.13-WP06 |
| --- | --- | --- |
| `entity=all`, `totalAuthority` | always `null` | `"snapshot"` / `"live"` / `null` |
| `entity=all`, banner | hidden | shown when any arm is `snapshot` |
| `entity=all`, `refresh_total` request | never fires | fires when user clicks |
| Single-arm behaviour | unchanged | unchanged |

The change is additive in the hook return shape (the type widens
from `"snapshot" | "live" | null` to the same set — values just
become reachable on `entity=all` instead of always-`null`). No
breaking change for consumers that read `totalAuthority`.

## Tests added

- Backend: `test_refresh_total_all_recounts_every_arm` (+1 net).
- Frontend hook:
  `WP06: totalAuthority on entity=all is "snapshot" when any arm
  is snapshot, "live" when every arm is live` (+1 net).
- Frontend page:
  `WP06: snapshot banner renders on the All tab when any arm is
  snapshot and Refresh counts fires refresh_total=true` (+1 net).

## Deferred

- Per-arm `refresh_total` opt-in syntax (option (a)) — not built,
  not needed today. Documented forward-door: change
  `SearchV2Params.refresh_total` from `boolean` to
  `boolean | string[]` and let the route accept a comma-list. The
  service layer would only need to gate each `_search_<arm>` call
  on arm-presence-in-list.
- "Show banner only after the user has been on the All tab for N
  seconds" — there's no signal source for staleness on entity=all
  (no cursor chain). The current "any-arm-snapshot" predicate is
  the simplest correct trigger.
