## 3. Plugin availability

### pydantic.mypy
- Direct `import pydantic.mypy` from a Python REPL **raises** at module load:
  `AttributeError: module 'mypy.expandtype' has no attribute 'ExpandTypeVisitor'`
- **However**, when the plugin is loaded via mypy itself (as opposed to a raw `import`), it works.
  The v2.19-WP02 diagnosis flagged the raw-import error and pre-emptively disabled the plugin.
  This recon shows mypy 1.20.1 + pydantic 2.13.0 actually load cleanly inside the mypy
  pipeline. The raw `import pydantic.mypy` failure is a red herring caused by mypy's vendored
  module shape, not by an actual plugin-load failure.

### sqlalchemy.ext.mypy
- Importable: ✅ (`/.venv/lib/python3.12/site-packages/sqlalchemy/ext/mypy/plugin.py`).
- **Deprecated** in SQLAlchemy 2.x; SQLAlchemy 2.0 with `Mapped[T]` annotations is designed to
  be plugin-less. Recommended path is "pure mode" + proper `Mapped[T]` annotations everywhere,
  not turning the plugin on.

### sqlalchemy2-stubs
- Not installed. Only useful for 1.4-style `Column(...)` patterns, which we are intentionally
  not adopting because the codebase has migrated to 2.0 `Mapped[T]` (incomplete in some
  models; that's the actual cluster source).

## 6. Recommended WP02 / WP03 packaging

### WP02 (recommended first): **enable `pydantic.mypy`**
- Lowest risk: 0 new errors in sandbox; only deletions from allow-list.
- Expected delta: 164 → 132 mypy errors (−32).
- Allow-list shrinks correspondingly. The 32 errors removed are not all in the
  pydantic-cluster (which is only 6 keys) — the plugin also unblocks downstream inference
  on classes that compose with Pydantic models, freeing some SQLAlchemy- and other-cluster
  entries too.
- Mechanical change: add `plugins = ["pydantic.mypy"]` under `[tool.mypy]` in
  `pyproject.toml`, then re-run `tests/test_typecheck_lint_v219_wp02.py`, regenerate the
  allow-list with the new (smaller) offender set, and re-record total / BY-DESIGN counts.
- Update the in-file comment block in `pyproject.toml` that currently says the plugin
  errors against mypy 1.20.1 — this is no longer true with pydantic 2.13.0.

### WP03 (recommended second): **leave sqlalchemy.ext.mypy OFF; target Mapped[T] coverage instead**
- The 136-entry SQLAlchemy cluster is dominated by `Column[T]` leakage past the ORM boundary.
- `sqlalchemy.ext.mypy` is deprecated in SA 2.x; turning it on would couple us to a
  sunset code path.
- `sqlalchemy2-stubs` only helps 1.4 patterns, not our 2.0 `Mapped[T]` shape.
- The structurally correct fix is **finishing the `Mapped[T]` migration** in the few
  `app/models/` files that still emit `Column[T]` types past the boundary, plus targeted
  `assignment`-cluster fixes in service code where `Column[bool] = True` patterns persist.
- WP03 shape proposal: audit `app/models/*.py` for any attribute still typed as bare
  `Column[T]` rather than `Mapped[T]`, plus a small sweep of the service-side
  `Incompatible types in assignment (expression has type "X", variable has type "Column[X]")`
  cluster (≥40 of the 136 SQLA entries per the head-of-list error tally). Budget this as
  source-code changes, not config changes. Plugin route is explicitly deferred / abandoned.

## 7. Risks / open questions

- The v2.19-WP02 diagnosis recorded the `ExpandTypeVisitor` import failure as the reason
  the plugin was disabled. Confirm whether the prior failure was on an earlier pydantic
  (pre-2.13.0) and the version bump fixed it, or whether v2.19 only tested the raw
  `import pydantic.mypy` path and never tried the plugin via mypy itself. Either way the
  current sandbox is clean.
- Updating the comment block in `pyproject.toml` is part of WP02 — leaving the stale
  "plugin errors against mypy 1.20.1" comment in would mislead future maintainers.
- `warn_unused_ignores = true` is on. The plugin sometimes makes existing
  `# type: ignore` lines redundant; the sandbox run did not report any new
  `unused-ignore` errors, but WP02 should re-verify against the live v2.17-WP03 inline-ignore
  allow-list.
