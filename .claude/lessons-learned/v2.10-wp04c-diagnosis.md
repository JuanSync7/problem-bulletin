# v2.10-WP04c — Side-effects live-DB port

## TL;DR

**51 deferred IDs ported to live-Postgres** across two side-effect
service test files:

- `tests/services/test_notifications.py` — 23 IDs (watch / notification
  generation / WebSocket push / Teams webhook / email digest)
- `tests/services/test_attachments.py` — 28 IDs (validate / store /
  list / delete)

All 51 are bucket **(c) — tests assert the wrong invariant**: kwargs
renamed (`current_user=`, `upload=`, `actor=`), helper signatures
drifted, and several v1 tests asserted *route-layer* invariants
(`HTTP 403`, `HTTP 404`) against the *service* layer that doesn't
enforce them. The notification-side drift was largely a fixture problem
— the watch / notification services were already correct, but the v1
tests fed them `MagicMock` watch rows whose `.user_id` happened to be a
fresh `uuid.uuid4()`, so the routing filters silently filtered
everything to one happy path.

**No production code bugs found** during the port. Bucket totals: **(a)
0 · (b) 0 · (c) 51**.

Full suite final: **1154 passed / 0 failed / 40 skipped / 14 xfailed**
(was 1103 / 0 / 91 / 14). `tests/_v1_deferred.py`: 86 → 35.

The remaining 35 deferred IDs are all in `tests/test_main.py` (32) and
`tests/test_schemas.py` (3) — both owned by **WP07**.

## Boundary mocking decisions

| Boundary | Patched at | Why |
| --- | --- | --- |
| Disk write | `app.services.attachments.store_file` | Cleanest seam — the function does the FS work *and* generates the UUID-filename, so patching it lets us assert the inputs (`file_bytes`, `problem_id`, `filename`) without an FS roundtrip. |
| Disk delete | `app.services.attachments._remove_file_from_disk` (or `pathlib.Path.unlink`) | Same reasoning. For the *disk-failure-logged* test we patch `Path.unlink` directly so the OSError surfaces inside `_remove_file_from_disk` and exercises its log-and-swallow path. |
| WebSocket | `app.services.delivery.connection_manager` | Avoids spinning a real ASGI test server. |
| Teams HTTP | `app.services.delivery.httpx.AsyncClient` | Forces a fake async context-manager that returns a stub response. |
| SMTP | `aiosmtplib.send` | Top-level send function — patching here is enough since `send_email_digest` calls `aiosmtplib.send(...)` directly. |
| Settings | `app.services.delivery.get_settings` | Only used in the "no-op when `TEAMS_WEBHOOK_URL=None`" test; everywhere else the real cached settings work fine. |

DB is always live (Postgres rolled-back per-test).

## v2.11 follow-ups discovered

1. **`delete_attachment` has no service-layer auth check.** The v1
   tests asserted 403 / 404 from the service, but those are raised by
   the route. If the contract is "service is auth-aware", lift the
   check (uploader-or-admin) into `delete_attachment` and replace the
   rewritten tests in WP04c with the original 403 / 404 assertions.
   If the contract is "service trusts its caller", document that on the
   function and add a `# noqa: auth-checked-in-route` style marker.

2. **`send_email_digest` early-return when `notifications == []`
   precedes the user lookup.** Production currently does
   `if not notifications: return` *before*
   `await db.execute(select(User).where(User.id == ...))`. So an empty
   list short-circuits the "user not found" path too. That's defensible
   (no point hitting the DB for an empty batch), but worth a comment
   so the contract is explicit.

3. **`send_teams_webhook` reads `settings.TEAMS_WEBHOOK_URL` (upper).**
   The v1 test patched the lower-case attribute and would have passed
   regardless. Consider adopting a single naming convention for
   settings (the rest of the codebase mostly uses upper-case).

4. **`set_watch` returns a `Watch` instance whose `.level` reflects the
   session's *first* materialisation of the row.** A second
   `set_watch(...)` upserts the row, but the returned object's
   `.level` attribute is still the original value until `await
   db.refresh(...)`. Reasonable behaviour for SQLAlchemy's identity
   map, but worth a docstring callout — the test added a `refresh()`
   to assert the post-upsert state. If callers want fresh-state
   semantics by default, `set_watch` should `db.refresh(watch)` before
   returning.

5. **`generate_notification` uses `select(Watch).where(... user_id !=
   actor_uuid)` and then iterates in Python to apply
   `WATCH_ROUTING`.** That's fine while the routing table is small
   and the watcher set per problem is bounded, but for a hot problem
   it spawns one Python loop per watcher. Could be tightened to a
   `WHERE level = ANY(:allowed_levels)` once `WATCH_ROUTING` is
   denormalised by event_type.

6. **`mark_notification_read / mark_all_notifications_read /
   get_notifications` skip via `ImportError` in the test file.**
   They've been "to be implemented" since v1. Either implement them
   (the routes use direct SQL today; consolidating into service
   functions would be a small win) or remove the skips and the
   imports they reference.

## Surprises

- **Zero production bugs.** WP04a was 0/91, WP04b was 1/77, WP04c is
  0/51. Total across the lifecycle / read-side / side-effects clusters:
  **1 real bug / 219 ports = 0.46% prod-bug rate**. The mock-DB tests
  were almost entirely noise once the contracts had drifted. Useful
  data point for "should we keep mock-DB tests?" debates.
- **The `_db_result` shim in the v1 file is genuinely incompatible
  with how `set_watch` reads back its returning row.**
  `result.scalar_one()` on a real result expects exactly one row;
  the mock returned the same `MagicMock` for both `.scalar_one` and
  `.scalar_one_or_none`. The mock-DB-friendly path was always a
  fiction.
- **SQLAlchemy's identity map made the `set_watch` upsert test look
  wrong.** First `set_watch(... solutions_only)` materialised the row
  with `.level == "solutions_only"`. The second call upserted the DB
  row to `all_activity`, but the cached Python instance still showed
  `solutions_only`. `await db.refresh(second)` resolves it. Worth
  noting because it's exactly the kind of subtle "the DB is right,
  the Python object isn't" failure that pure-mock tests can never
  catch — and that the live-DB port surfaces immediately.
- **`_remove_file_from_disk` catches OSError internally.** Patching
  it at the helper-function level prevents the failure path from
  surfacing in `delete_attachment`. The test had to patch
  `pathlib.Path.unlink` instead. Documented in the boundary table
  above and in the test docstring.
- **`generate_notification`'s actor exclusion is done in SQL
  (`Watch.user_id != actor_uuid`), not in Python.** So even a
  malformed routing table couldn't accidentally notify the actor.
  Nice belt-and-braces in production code; the v1 mock test was
  asserting the right thing for the wrong reason (it didn't get to
  the routing check at all because the actor was the only watcher).
