# v2.11-WP06 — Page[T] adoption across ad-hoc paged-list routes

**Status**: ✅ landed
**Pytest**: 1231 passed / 0 failed / 5 skipped / 14 xfailed (was 1212 / 0 / 5 / 14)
**Bucket**: B1 (response-shape convergence)

## Goal

Converge the seven ad-hoc paged-list route handlers that returned raw
`dict[str, Any]` onto the canonical `app.schemas.common.Page[T]` envelope
so OpenAPI advertises a typed response_model, FastAPI validates outbound
payloads, and the frontend has a single envelope contract to parse.

## Page[T] vs current dict shape

Confirmed `Page[T]` fields (`app/schemas/common.py`):

```
items:        list[T]
next_cursor:  str | None  = None
total:        int | None  = None
```

Pre-WP06 dict shapes at the seven call sites:

| File:line | Pre-WP06 dict keys | Match? |
|---|---|---|
| `app/routes/agents.py:97` (+ compat:110) | `items, limit, offset` | **drift** — limit/offset extra, next_cursor/total missing |
| `app/routes/projects.py:83`  | `items, next_cursor, total` | exact |
| `app/routes/projects.py:154` | `items` only | subset |
| `app/routes/projects.py:228` | `items` only | subset |
| `app/routes/sprints.py:51`   | `items` only | subset |
| `app/routes/tickets.py:249`  | `items` only | subset |
| `app/routes/tickets.py:707`  | `items` only | subset |
| `app/routes/tickets.py:775`  | `items` only | subset |

## Decision-gate verdict — PROCEED

The only "real" drift is `agents/activity`, which currently leaks
`limit`/`offset` into the response body. I confirmed it is safe to drop
those keys:

- `frontend/src/api/audit.ts` reads `body.items` only — `limit`/`offset`
  are query params, never response body fields.
- `tests/routes/test_agents_activity.py` only asserts `body["items"]`
  presence/contents — no `body["limit"]`/`body["offset"]` references.
- Global grep `body\["limit"\]|body\["offset"\]` across `tests/`
  returned zero hits.

For the other six call sites the conversion is purely additive
(`next_cursor`/`total` added, nothing removed) — frontend safely ignores
unknown keys.

## Per-route conversion table

| File:line | Before | After | Tests touched |
|---|---|---|---|
| `app/routes/agents.py` `list_activity` / `list_activity_compat` | `dict[str, Any]` returning `{items, limit, offset}` | `response_model=Page[AgentActivityItem]`, returns `Page[AgentActivityItem](items=..., next_cursor=None, total=None)` | none (existing assertions still hold) |
| `app/routes/projects.py` `list_projects` | dict `{items, next_cursor, total}` | `response_model=Page[ProjectRead]` | none |
| `app/routes/projects.py` `list_members` | dict `{items}` | `response_model=Page[ProjectMemberRead]`, populates `total=len(items)` | none |
| `app/routes/projects.py` `list_components` | dict `{items}` | `response_model=Page[ComponentRead]`, populates `total=len(items)` | none |
| `app/routes/sprints.py` `list_sprints` | dict `{items}` | `response_model=Page[SprintRead]`, populates `total=len(items)` | none |
| `app/routes/tickets.py` `search_tickets` | dict `{items}` | `response_model=Page[TicketRead]`, `total=None` (search has no cheap COUNT) | none |
| `app/routes/tickets.py` `list_watchers` | dict `{items}` | `response_model=Page[TicketWatcherRead]`, `total=len(items)` | none |
| `app/routes/tickets.py` `list_attachments` | dict `{items}` | `response_model=Page[TicketAttachmentRead]`, `total=len(items)` | none |

`total` policy: where a single in-process `len(items)` is exact (small
collections already materialised), we set `total`; for unbounded
queries (`tickets/search`, `agents/activity` offset paging) we leave it
`None` to avoid implying an authoritative count the service didn't
compute. Matches `Page[T]` docstring guidance.

## New schemas added

In `app/schemas/tickets.py`:

- `TicketWatcherRead` — mirrors `TicketWatcher.to_dict()`.
- `TicketAttachmentRead` — mirrors `TicketAttachment.to_dict()`.
- `AgentActivityItem` — projection over `audit_log_event` rows used by
  `/api/agents/activity`. Fields mirror the legacy `_list_activity`
  dict (id, occurred_at, actor_id, actor_type, actor_name, action,
  entity_type, entity_id, ticket_key, correlation_id, details).

`ProjectRead`, `ProjectMemberRead`, `ComponentRead`, `SprintRead`,
`TicketRead` were already defined; no new project/sprint schemas
required.

## Frontend-shape risk

**No breaking change**, with one approved drift:

- `agents/activity`: response no longer includes `limit`/`offset`.
  Frontend (`audit.ts`) doesn't read them. Treat as a benign cleanup.
- Six other routes: response GAINED `next_cursor=null` and `total=<int>`.
  Frontend reads `body.items` only — additive keys are ignored.

## Out-of-scope (Bucket B2 / future WPs)
