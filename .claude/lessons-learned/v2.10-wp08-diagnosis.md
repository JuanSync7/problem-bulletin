# v2.10-WP08 — Wire cursors through Search UI

Wire the HMAC-signed cursor pagination (delivered backend-side in v2.9-WP62)
through the Search page's React layer so deep paging no longer relies on
offset arithmetic.

## TL;DR

- Backend `/api/search/v2` already accepts a `cursor=` param (single-arm) and
  per-arm `<arm>_cursor=` params (entity=all) and returns `next_cursor` on
  every arm. The contract is intact and was never plumbed through the API
  client or the `useSearchV2` hook.
- API client and hook now pass `cursor` end-to-end on single-arm tabs.
- Hook maintains a per-arm cursor stack so Prev navigation works without
  re-querying older pages from offsets.
- UI replaces the offset Prev/Next page-number widget on single-arm tabs
  with cursor-driven Next/Prev. The "All" tab is unchanged (preview slice).

## G1 — Audit (current behaviour)

`frontend/src/pages/Search.tsx`
- Maintains `page: number` state, reset to `0` on query change / tab change /
  filter change.
- Passes `page`, `pageSize` to `useSearchV2`.
- Renders `<ArmView>` with offset-style pagination: "Page X of Y", Prev/Next
  buttons gated on `page` and `Math.ceil(total/pageSize)`.

`frontend/src/hooks/useSearchV2.ts`
- Computes `offset = entity === "all" ? 0 : page * pageSize` and passes to
  `searchV2()`.
- No cursor awareness whatsoever.

`frontend/src/api/search.ts`
- `SearchV2Params` exposes `limit` + `offset` only.
- `SearchArm` shape does **not** declare `next_cursor`, even though backend
  emits it.
- `searchV2()` URL builder has no branch for any cursor param.

Backend (`app/routes/search.py`, `app/services/search_multi.py`)
- Confirmed: `/api/search/v2` accepts `cursor=` (single-arm) and
  `{problems,tickets,components,labels,users}_cursor=`. `cursor=` is only
  valid when `entity` is one of the five arm names (HTTP 400 otherwise).
- Each arm result dict includes `next_cursor: str | None`. Cursor is HMAC-
  signed via `_pagination.encode_signed_cursor`. Invalid cursors → HTTP 400.

## G2 — Design choice

**Cursor-only Next/Prev for single-arm tabs (no page numbers, no jumps).**

Rationale:
- Backend contract is seek-pagination on `(rank, created_at, id)` per arm —
  it has no notion of "page N", so any page-number UI is a fiction layered on
  top of offset queries which the backend now serves with a different code
  path on cursor=. Mixing the two is confusing and incorrect for any data
  that changes mid-paging.
- Hybrid (offset for first N pages then cursor for deep scrolling) doubles
  the surface area and forces both API client and hook to handle two modes.
  WP10 in the v2.10 roadmap already calls out "stable-total mode for cursor
  pagination"; that's the place to revisit if we want totals + deep paging.
- Cursor-only matches the search use case: users explore the top of the
  rank distribution and rarely scroll past page 2. A Next/Prev pair is
  enough.
- Prev navigation uses a **client-side cursor stack**: each fetch pushes the
  cursor used (or `null` for page 1) onto a stack; Prev pops twice (current,
  desired) and re-issues with the popped cursor. No backend prev_cursor
  required.

Out of scope for WP08 (tracked in v2.11 backlog):
- "All" tab cursor wiring — preview slice is fine on first page.
- URL sync for cursor (cursors are opaque + HMAC-signed; round-tripping them
  through the URL bar is noisy and the next/prev pair is sufficient).
- Stable-total mode (WP10).
- v1 `/api/search` cursor parity (WP09).

## G3 — Backend contract

No backend changes. Only the frontend API client gains a `cursor?: string`
field on `SearchV2Params`, forwarded as `cursor=` on the query string when
present. The `next_cursor` field on `SearchArm` is added to the TS types so
the hook can read it without `unknown` casts.

## G4 — Hook update

`useSearchV2` gains:
- Internal `cursorStackRef: string[]` — entries are the cursor used to fetch
  each historical page (the first entry, page 1, is the empty string `""`).
  Using a ref (not state) avoids a re-render race when stack mutations happen
  inside the fetch effect.
- `loadNext()`, `loadPrev()` callbacks returned from the hook.
- `hasNext: boolean` (true when `data.<activeArm>.next_cursor` is set).
- `hasPrev: boolean` (true when the stack has >1 entry).

The existing `data` / `isLoading` / `error` / `hasSearched` fields are
preserved bit-for-bit. Existing callers that don't reference cursor fields
keep working. `page` and `pageSize` arguments are retained for the "All"
tab and as a fallback so this WP doesn't touch the other consumers of the
hook (there aren't any today, but the contract stays open).

The stack resets when any of `query` / `entity` / a filter changes.

## G5 — UI wiring

`Search.tsx` `<ArmView>` is replaced with cursor-driven Next/Prev:
- `hasPrev` enables Prev; `hasNext` enables Next.
- Removes "Page X of Y" — replaced with "Showing N of T results" where T is
  the arm `total` (backend continues to return total alongside cursor).
- Existing offset-style "page" state is dropped on single-arm tabs and only
  used for the "All" preview limit (which never paginates).

## G6 — Tests (RED-first)

Unit (`useSearchV2.test.ts`):
- `loadNext()` re-fires `searchV2` with `cursor=<next_cursor>` of the active
  arm.
- `loadPrev()` re-fires `searchV2` with the previous stack entry's cursor
  (or no cursor for page 1).
- Changing `query` resets the stack — a subsequent `loadPrev()` is a no-op /
  `hasPrev=false`.

Integration (`Search.test.tsx`):
- After receiving a tickets response with `next_cursor`, the Next button is
  enabled. Clicking it fires `searchV2` with `cursor=<that value>` and
  `entity=tickets`.
- Prev is disabled on page 1; enabled after Next; clicking returns to the
  cursor-less initial fetch.
- New query input resets pagination — Next is disabled until a fresh
  response returns `next_cursor`.

## G8 — Outcomes / surprises

- **State-vs-ref dance.** First implementation stored the cursor stack in a
  ref and a separate `cursorVersion` state to force re-renders. The fetch
  effect's dep list contained `cursorVersion` *and* `query`, which caused
  the args-change reset effect to fire a useless extra fetch after a query
  change. Fixed by:
  1. Moving the stack into `useState`.
  2. Putting the "args drifted → reset stack" logic *inside the fetch
     effect itself*, so an args change triggers the reset and the
     subsequent fetch in one render pass (not two).
  3. Bailing out of the first effect-pass when the reset was scheduled, so
     the second pass (with the freshly-reset stack) is the one that fires.
- **Mock-cache trap.** `mockResolvedValueOnce` chains in vitest will return
  `undefined` once exhausted, which surfaces as "expected undefined to
  deep-equal { … }" rather than "called too many times". This wasted ~5
  minutes; lesson is to set a default `mockResolvedValue` as the safety
  net after the `Once` chain.
- **"All" tab is unchanged.** `hasNext` / `hasPrev` are forced false when
  `entity === "all"` — the All tab preview slice is intentionally
  non-paginated; deep paging belongs on a single-arm tab where the
  per-arm cursor is meaningful.
- **CSS reused.** No new CSS — the existing `search-v2-pagination` /
  `search-v2-pagination__btn` styles still apply (Prev + Next, sans page
  counter).
- **`page` state still threaded through hook.** Kept for back-compat with
  the hook's `UseSearchV2Args` shape; the hook only consults it on
  fallback (when no cursor is in play). The Search page still resets
  `setPage(0)` on filter change for the same reason, even though it's now
  effectively a no-op for cursor-paginated arms. Cleanup deferred until a
  second consumer of `useSearchV2` lands.
