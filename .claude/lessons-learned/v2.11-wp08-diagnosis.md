## Iterations

### Iteration 1: naive prefix-import resolver false-positived

First implementation of C1 split `a.b.c.d` on the **last** dot, treating `a.b.c` as the prefix to import. That broke on:

| Hit (initial-run FAIL) | Why naive resolver flagged | Why it's actually fine |
| --- | --- | --- |
| `patch("pathlib.Path.unlink")` | `import pathlib.Path` → `ImportError: not a package` | `Path` is a class on `pathlib`, not a submodule. `mock` walks attributes. |
| `patch("app.services.delivery.httpx.AsyncClient")` (×3) | `import app.services.delivery.httpx` → fails | `httpx` is re-exported as an attribute on `app.services.delivery`; not a submodule. |

These are **legitimate** mock patterns — `unittest.mock` resolves dotted targets by importing the **longest** importable module prefix and then walking the rest as attribute access.

### Fix

Replaced `_resolve_dotted` with the same algorithm `mock` uses:

1. Walk back from longest prefix (`parts[:-1]`) down to shortest (`parts[:1]`), `importlib.import_module` each, stop on first success.
2. From that imported module, walk the remaining `parts` as `getattr` hops.
3. Any missing attribute → fail with the exact `walked-so-far` and the missing leaf in the message.
4. No importable prefix at all → fail with the last `ImportError`.
5. Import-time side-effect crash (anything other than `ImportError`) → SKIP, not fail (we can't tell whether the target is bad or the env is).

After the fix, the four previous "hits" all resolve cleanly, and the synthetic-bad self-test still catches a truly-missing attribute.

## Exclusion rules (per goal G4)

### C1 — patch-string resolution

| Excluded pattern | Reason |
| --- | --- |
| `patch.dict("os.environ", {...})` | Different mechanism (mapping mutation); first arg isn't a dotted-symbol path. Detected by checking for `Attribute(attr="dict")` whose `.value` is `patch` (or `*.patch`). |
| `patch.object(real_obj, "attr")` | First arg is a real object, not a string — falls through the `Constant(str)` filter naturally. |
| First arg with no `.` | Single-name strings aren't `module.symbol`. (Also covers `patch.dict("FLAG", ...)` style.) |
| f-string first arg (`JoinedStr`) | Dynamic target — can't statically resolve. |
| Non-string first arg (`Name`, `Call`, etc.) | Out of static scope. |
| `__SKIP__` import-time side-effect crash | Env issue, not a target issue. Logged but doesn't fail the test. |
| This file itself (`test_patch_string_resolution_lint_wp08.py`) | Contains synthetic-bad patch strings in the self-test; would self-trigger. |

### C2 — text bind-cast

| Excluded pattern | Reason |
| --- | --- |
| Comments and docstrings | Don't reach `text(...)` `Call.args[0]` via the AST surface — naturally excluded by the AST-targeted approach. |
| Non-string args (variables, runtime concatenations) | Out of static scope. |
| Naked `column::int` cast (no leading-colon bind name) | Legitimate PostgreSQL cast. The regex `:[a-zA-Z_][a-zA-Z0-9_]*::[a-zA-Z]` anchors on `:bind::`, not on `::` alone. |
| `CAST(:bind AS int)` form | The correct alternative; regex doesn't match. |

## v2.11 follow-ups

1. **C1 false-negative awareness:** the resolver only catches `patch("string-literal", ...)` forms. `patch(MY_CONST, ...)` where `MY_CONST` is a module-level string variable would be missed. If this pattern shows up, extend the scanner to resolve simple module-level `Assign(Constant(str))` aliases.
2. **C2 false-negative awareness:** string-concatenation builds of `text(...)` args (`text("WHERE " + clause)`) won't be inspected statically. If/when a real bug surfaces in concatenated SQL, consider a runtime-side guard rather than static.
3. **Reusable shape:** these are the third (after WP02 `legacy_status`) "AST scan of literal contents" pytest-test lints. If a fourth lands, consider factoring `_iter_*_py_files` / string-literal harvesting into a `tests/_lint_helpers.py`.
