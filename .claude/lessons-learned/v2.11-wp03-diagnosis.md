# v2.11-WP03 — admin service input validation + audit-actor consistency

## Bucket A3 + A4 — combined

Two admin-service-layer fixes sharing the same test surface
(`tests/services/test_admin.py`).

---

## A3 — `update_user_role` role-string validation

**Diagnosis.** The pre-WP03 implementation wrote whatever string the
caller passed straight onto `User.role`. Only the route's pydantic body
schema (`RoleUpdate(role: str)` — actually still untyped at WP03) sat
between callers and the DB; any non-route caller (background job, MCP
tool, sibling service) could persist garbage values like
`"totally_invalid"` and silently break role-based authorization.

**Choice.** Tightened the service. The canonical role set is
`app.enums.UserRole = {"user", "admin"}`; we now source `_ALLOWED_ROLES`
directly from that enum so the service can never drift from the schema.

**Failure mode if violated.** Domain `ValidationError` (matches the
WP01 `created_by` guard shape, `fields=[{"name": "role", "reason": ...}]`).

## A4 — audit-actor consistency

**Convention pinned.** `log_event(...)`'s `user_id` slot is the
**principal id of the caller** (the actor performing the change), NOT
the target entity id, NOT a literal label.

### Backward-compat note

`update_user_role`, `update_user_status`, and `update_config` all accept
`actor_id` as a keyword-only optional argument with a safe fallback so
existing tests that don't thread an actor still pass:

- `update_user_role` / `update_user_status` — fall back to target
  `user_id` (the prior behaviour, documented as legacy).
- `update_config` — falls back to the literal `"system"` (clearly
  non-user). Previously this slot held `"admin"`, which collided with
  the role name and was actively misleading.

Routes that ARE auth-aware (admin routes) MUST thread the admin's id
into `actor_id`; this is enforced at the route layer (see G5 below).

## v2.11 follow-ups

- `update_user_role` / `update_user_status` still accept a `None`
  `actor_id` for backward compatibility, falling back to target id.
  When all callers are confirmed to thread actor explicitly, make
  `actor_id` required and remove the fallback.
- The route schema `RoleUpdate(role: str)` could be tightened to
  `role: Literal["user", "admin"]` (or `UserRole`) for parity with the
  service validator. Defensive duplication, but yields a 422 (pydantic)
  before the request even reaches the service. Park as a small cleanup.
- `log_event` has no compile-time type for the `user_id` slot — drift
  was the root cause here. Consider tightening to `str | UUID` and
  documenting the "caller principal id" convention on the helper
  docstring.
