## OpenAPI type shapes

The backend already exposes concrete response schemas (the v2.17 allow-list
comment cited "EditSuggestionRead / AttachmentRead" — these are aspirational
names for the response models). No frontend `*Read`-named type existed; the
api/ directory carries only `*DTO` interfaces. I introduced two file-local
interfaces mirroring the backend pydantic shapes 1:1, named per the
allow-list's prescriptive language so future readers grep cleanly.

### `EditSuggestionRead` (mirrors `app/routes/edit_suggestions.py::EditSuggestionResponse`)

```ts
interface EditSuggestionRead {
  id: string;
  problem_id: string;
  author: Author | null;          // backend returns dict|None — Author shape is { id, display_name }
  suggested_description: string;
  reason: string | null;
  status: string;                  // "pending" | "accepted" | "rejected"
  created_at: string;              // ISO timestamp
}
```

### `AttachmentRead` (mirrors `app/routes/attachments.py::AttachmentResponse`)

```ts
interface AttachmentRead {
  id: string;
  parent_type: string;             // ParentType enum value
  parent_id: string;
  uploader_id: string;
  filename: string;
  content_type: string;
  byte_size: number;
  storage_path: string;
  render_inline: boolean;          // derived in `_to_response`: content_type.startsWith("image/")
  created_at: string;              // datetime serialized as ISO string by FastAPI
}
```

## Consumer-side adaptation

No narrowing was needed at the `.map` callsites:

- `editSuggestions.map((s) => …)` reads `s.id`, `s.author?.display_name`,
  `s.reason`, `s.suggested_description` — all match `EditSuggestionRead`.
  Existing `s.author?.display_name ?? "Anonymous"` already null-safed the
  optional author dict, so the `Author | null` typing is exact.
- `attachments.map((att) => …)` reads `att.id`, `att.content_type`
  (optional-chained as `att.content_type?.startsWith("image/")`),
  `att.filename`. `att.content_type` is non-optional in the backend
  schema; the optional-chain in the consumer is defensive but harmless.

`setEditSuggestions(await res.json())` and `setAttachments(await res.json())`
compile because `Response.json()` returns `Promise<any>` in lib.dom.d.ts —
the unsafe assignment crosses the network boundary unchanged, which is the
same pattern the rest of the file uses (e.g. `setProblem`, `setSolutions`).
A future sweep could thread these through `unknown` + a runtime parser,
but that's out of scope for the WP02 type-tightening.
