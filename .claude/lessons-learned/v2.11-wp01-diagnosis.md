# v2.11-WP01 — agent_accounts.created_by ORM↔DB alignment

## Context

v2.10-WP02 closed the **data-layer** drift on `agent_accounts.created_by` by
(a) adding migration `a17` to tighten the column to `NOT NULL` and
(b) routing all test seeding through `tests/helpers/seed_agent_account.py`.
What it did **not** do: tighten the ORM column or the service signature.

`app/models/agent_account.py` still declares:

```python
created_by: Mapped[UUID | None] = mapped_column(
    PgUUID(as_uuid=True), ForeignKey("users.id"), nullable=True,
)
```

and `AgentAccountService.create_account(..., created_by: UUID | None = None)`
still accepts `None`. Result: a future caller that forgets `created_by`
type-checks clean and only fails at the DB layer with an `IntegrityError`,
which is much harder to debug than a clean validation error.

This WP closes that gap at the type-checker / service boundary.

## Validatable end goals

- **G1**: New test asserts `AgentAccount.__table__.columns['created_by'].nullable is False`. RED on main, GREEN after fix.
- **G2**: New test asserts that `AgentAccountService.create_account(...)` called without `created_by` (or with `created_by=None`) raises a clear `ValidationError` **before** the DB write attempt. RED on main (raises `IntegrityError` instead), GREEN after fix.
- **G3**: `app/models/agent_account.py` `created_by` → `nullable=False`, type tightened to `UUID` (not `UUID | None`). Service signature tightened to require `created_by: UUID` (no default). Mirror tightening in any pydantic shape if one exists.
- **G4**: Backend suite remains green: ≥1190 passed + the new tests, 0 failed, prior skipped/xfailed counts preserved.
- **G5**: No new alembic migration. DB already enforces NOT NULL via `a17`.

## Failing test plan

Two tests, added to `tests/test_agent_accounts_created_by_orm_alignment.py` (a new file beside the existing `tests/test_agent_accounts_created_by.py` from v2.10-WP02, so the port-in-place rule isn't violated):

1. `test_agent_account_model_created_by_is_not_null` — `__table__` introspection.
2. `test_service_create_account_without_created_by_raises_validation_error` — calls `create_account` without `created_by` and expects `ValidationError`, not `IntegrityError`.

Both fail on unchanged main:
- G1 fails because the column is `nullable=True`.
- G2 fails because the service flushes to DB and raises `IntegrityError` (or `TypeError` after we drop the default but before adding the validation guard, depending on order).

## Production change

In `app/models/agent_account.py`:

```python
created_by: Mapped[UUID] = mapped_column(
    PgUUID(as_uuid=True), ForeignKey("users.id"), nullable=False,
)
```

In `app/services/agent_accounts.py`:

```python
async def create_account(
    self,
    session: AsyncSession,
    *,
    name: str,
    created_by: UUID,
    scopes: Sequence[str] | None = None,
    description: str | None = None,
) -> tuple[AgentAccount, str]:
    if created_by is None:
        raise ValidationError([{"name": "created_by", "reason": "required"}])
    ...
```

`created_by` becomes required (no default) and there's an explicit guard that
raises `ValidationError` before the DB flush. The guard matches the existing
empty-name pattern.

No pydantic schema for `AgentAccountCreate` exists in
`app/routes/admin/agent_accounts.py` that mirrors `created_by` (the route
synthesises it from `actor.id`), so no schema tightening is needed.

## Caller seeds touched

- `app/routes/admin/agent_accounts.py` already passes `created_by=actor.id`.
- All `tests/services/test_agent_account_service.py` tests already seed a
  user and pass `created_by=creator` after v2.10-WP02.
- The existing `tests/test_agent_accounts_created_by.py::test_service_create_account_without_created_by_raises_integrity_error` previously asserted `IntegrityError`. After the fix it would assert `ValidationError`. Per port-in-place, this is a deferred-test of exactly the failure mode WP01 is changing, so an update is justified: I'll change the expected exception to `ValidationError` and update the docstring to reflect the new contract (service-layer guard rather than DB constraint).
- `tests/services/test_ticket_notifications_wp34.py::_FakeAgentNullOwner.created_by = None` is a mock simulating pre-a17 state in-memory; it does **not** go through the ORM/DB and therefore is unaffected.
- `scripts/create_agent_account.py` and `scripts/agent_demo_direct.py` — verified they pass `created_by` explicitly.

## Lessons / v2.11 follow-ups

- v2.10-WP02 closed the DB drift but left the ORM type loose — a useful
  reminder that "drift" can hide on either side of the ORM↔DB seam.
- Worth a future Bucket-A sweep: grep for `nullable=True` in
  `app/models/*` and cross-check each against the corresponding migration
  history for similar drifts (e.g. `last_seen_at`, `revoked_at` are
  expected to remain nullable but should be confirmed).
- Service signature was using `created_by: UUID | None = None` purely as a
  convenience default; making it kw-only required is the right boundary.
