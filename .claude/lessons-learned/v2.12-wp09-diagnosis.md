## 2. Modules that previously registered `add_exception_handler`

Only one production module did so: `app/main.py::create_app`. It:

1. Registered `@app.exception_handler(AppError)` for the central
   `_EXCEPTION_STATUS_MAP` (legacy `{"detail": ...}` shape).
2. Registered `@app.exception_handler(...)` for each of the four
   `services/exceptions` classes (also `{"detail": ...}`).
3. Loop-registered every entry from
   `app.routes.tickets.EXCEPTION_HANDLERS` (the unified
   `{"error": {...}}` shape).

`app/routes/tickets.py` does NOT itself call `add_exception_handler` —
it exports the `EXCEPTION_HANDLERS` dict and lets `create_app` mount
them. No other route module registers handlers.

## 3. Envelope shapes BEFORE WP09

- `{"detail": "..."}` — every `AppError` subclass routed through the
  central handler, every `services/exceptions` class, every
  FastAPI-default `HTTPException`, every Pydantic `RequestValidationError`.
- `{"detail": "...", "next_allowed_at": "..."}` — bespoke shape for
  `HandleChangeTooSoonError` (429).
- `{"error": {"code", "message", "details", "correlation_id"}}` — every
  exception listed in `app/routes/tickets.py::EXCEPTION_HANDLERS`.

## 4. Envelope AFTER WP09

Every error-response path emits the unified shape:

```json
{"error": {
    "code": "<machine_code>",
    "message": "<human_msg>",
    "correlation_id": "<id>" | null,
    "details": {...} | null
}}
```

The `code` field is either:

- the per-class code declared in `app/routes/tickets.py::EXCEPTION_HANDLERS`
  (rich domain codes: `not_found`, `conflict`, `invalid_transition`,
  `already_claimed`, `link_exists`, `forbidden`, `validation`,
  `unauthorized`); or
- the entry from `app/main.py::_EXCEPTION_CODE_MAP` for plain-AppError
  subclasses (`pin_limit_exceeded`, `duplicate_vote`, `file_size_limit`,
  `file_type_not_allowed`, `magic_link_expired`, `tenant_mismatch`,
  `not_found`); or
- `http_error` for any `HTTPException` not caught earlier (preserves
  the original `detail` string as the envelope `message`); or
- `validation` for any Pydantic `RequestValidationError`; or
- `forbidden` / `handle_taken` / `handle_change_too_soon` /
  `profane_handle` for the four `services/exceptions` classes.

`correlation_id` is read from
`app.middleware.logging._correlation_id_ctx` (populated by
`CorrelationIdMiddleware` on every request) — NOT from the response
header (which is set on the way out and isn't readable from a
handler). When the contextvar is empty, the envelope serialises
`null` rather than fabricating an id.

## 6. Deferred to v2.13

- Sweeping audit of every frontend fetch call site to route through
  `parseApiError`. WP09 explicitly limited scope to 2-3 high-impact
  sites; ~20+ remain (search.ts, tickets.ts, notifications.ts,
  projects.ts, people.ts, sprints.ts, etc.). Each currently parses
  errors ad-hoc; they will still work since the backend `message`
  text remains useful, but consistent `code`-based UI error handling
  needs a sweep.
- WebSocket error paths (`app/routes/realtime_ws.py`,
  `app/routes/ws_tickets.py`) — out of scope per WP09 invariant 5.

## 7. Backwards compatibility

Zero — the app has no public consumers outside this repo. The
frontend adapter accepts both shapes only as a transitional safety
net for any path we haven't migrated yet.
