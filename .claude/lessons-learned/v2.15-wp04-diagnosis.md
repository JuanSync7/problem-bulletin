# v2.15-WP04 (C3) ‚Äî OTel test-mode noise quieting

## Goal

Eliminate two stderr-noise patterns leaked during pytest runs that have been
on the baseline since v2.14-WP01 (and re-confirmed in v2.15-WP01):

1. **`ConnectionRefused` / `localhost:4317`** ‚Äî gRPC export failures when no
   OTLP collector is running.
2. **`ValueError: I/O operation on closed file`** ‚Äî `ConsoleMetricExporter`
   writing to a stdout stream that pytest has already torn down.

Constraint: do **not** alter production behaviour and do **not** break
`tests/observability/test_otel_init.py` assertions (those tests verify the
*span* exporter wiring for both empty-endpoint and `localhost:4317` cases).

```

    self._exporter.export(
  File ".../opentelemetry/sdk/metrics/_internal/export/__init__.py", line 176, in export

```

```
to remote host: Connection refused"
```

Both originate in `PeriodicExportingMetricReader`'s **background ticker
thread** ‚Äî which fires every export-interval and survives the test setup that
created it. Span exporters are batch-only and do not periodically export, so
they were silent in this environment.

## Test-mode detection idiom

`os.environ["PYTEST_CURRENT_TEST"]`. This is pytest's standard runtime
sentinel ‚Äî set automatically while any test is collecting/executing/tearing
down. No existing project-level idiom (`TESTING`, `ENV=test`, etc.) was
present in `app/` or `tests/` (`grep -rn "PYTEST_CURRENT_TEST\|TESTING\|in_test_mode"`
returned zero hits before this WP), so this is a fresh introduction kept
localised to `app/observability/otel.py` via a private `_in_test_mode()`
helper.

Chosen over alternatives:

* `settings.ENVIRONMENT == "test"` ‚Äî the test conftest pins `ENVIRONMENT` to
  `development` deliberately (WP05 fail-fast guard) so it cannot signal test mode.
* Custom `TESTING=1` env var ‚Äî would require touching conftest; not worth it
  when pytest sets one for free.

## Fix path chosen ‚Äî Path A (in-memory metric reader)

Swap `PeriodicExportingMetricReader(<Console|OTLP>MetricExporter)` for
`InMemoryMetricReader` when `_in_test_mode()`. This:

* Removes the periodic background thread ‚Üí no more late-firing `Console`
  writes after stdout closes ‚Üí no more `I/O operation on closed file`.
* Removes the gRPC client that retries against `localhost:4317` ‚Üí no more
  `ConnectionRefused` tracebacks.
* Leaves *span* exporters (Console/OTLP) untouched ‚Üí existing assertions in
  `test_otel_init.py` that introspect `_active_span_processor._span_processors`
  for `ConsoleSpanExporter` / `OTLPSpanExporter` continue to pass.
* Leaves production wholly untouched ‚Äî the test-mode branch is only entered
  when `PYTEST_CURRENT_TEST` is set.

Path B (raise OTel logger levels) was rejected because:

* It would not address the `I/O operation on closed file` traceback ‚Äî that
  comes from `logging`'s own `StreamHandler.emit`, downstream of the OTel
  logger, so raising the OTel logger's level doesn't help once the export
  call itself touches the dead stream.
* It would mask production diagnostics if accidentally extended.

## Regression test

`tests/observability/test_otel_silence_wp04_v215.py` ‚Äî
`test_otel_setup_emits_no_stderr_noise_under_pytest`. Subprocess-runs
`pytest tests/observability/test_otel_init.py -q` (the canonical surface
that exercises both empty-endpoint and `localhost:4317` paths), captures
stdout+stderr, and asserts none of `localhost:4317`, `ConnectionRefused`,
`Connection refused`, or `I/O operation on closed` appear. Confirmed RED on
the pre-fix tree (manual inspection ‚Äî these were the baseline patterns) and
GREEN post-fix.

Subprocess scope is intentionally narrow (one file, ~0.4s) so the regression
test runs fast.
