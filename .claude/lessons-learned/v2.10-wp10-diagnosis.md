# v2.10-WP10 — Stable-total mode for cursor pagination

## Problem
Cursor pagination on `/api/search/v2` returns `total` per arm via
`COUNT(*) OVER ()` window. That count is computed at request time, so inserts
or deletes between requests cause the value to drift mid-scroll. UI shows
"Showing X of 100" then jumps to "Showing X of 102" between pages.

## Goals
- **G1 — Audit current behavior**: pending → done
- **G2 — Design choice**: embed total in HMAC-signed cursor payload (option (i))
- **G3 — Implementation**: extend `_build_next_cursor` to thread snapshot total; on cursor input, read embedded `total` and skip the `COUNT(*) OVER ()` work where possible (or override with embedded value)
- **G4 — Backward compat**: cursors lacking the `t` (total) field still decode → fall back to live count
- **G5 — Tests**: cursor roundtrip + insert-between-pages stable + backward-compat
- **G6 — Frontend impact**: `useSearchV2` consumes `total` opaquely; no UI change required
- **G7 — Suite green**: ≥1186 backend + 229 frontend
- **G8 — Diagnosis doc + ticketing-v2.10.md WP10 section**

## G1 — Audit (current behavior)
- Route: `app/routes/search.py::search_v2` — orchestrates arm dispatch and cursor decoding.
- Service: `app/services/search_multi.py::_search_<arm>` — each arm runs a CTE wrapping the hit-producing query, then `SELECT ..., COUNT(*) OVER () AS total_count` and applies seek/offset.
- `total` returned to the client is read from `rows[0]["total_count"]` on every page.
  - On page 1 (no cursor) and page N (with cursor), the same `COUNT(*) OVER ()` runs against the live hits CTE — any insert that produces a hit increases the count; any delete decreases it.
- Drift surface: page 1 returns total=100. A row matching the query is inserted. Page 2 (cursor) returns total=101. UI counter jumps. Same applies in reverse on deletion.

## G2 — Design
Pick **option (i)**: embed `total` (an integer snapshot) inside the HMAC-signed
cursor payload as field `t`. Rationale:
- Cursor is already HMAC-signed → total cannot be tampered.
- No new query param required; matches WP62 envelope shape.
- Frontend (WP08 `useSearchV2`) treats cursor + total opaquely; zero UI churn.
- Cheap to add — payload gains one int field; backward-compat for legacy
  cursors that lack `t` (fall back to live count).
- Bonus: subsequent pages can use the embedded total directly without
  re-running COUNT — this is now considered an optimisation. We still let SQL
  return `total_count` (we need rows anyway), but the **returned** `total` is
  pinned to the cursor snapshot, so even if the live count drifts, the API
  output is stable for the scroll session.

Cursor format change: **additive field** `"t": <int>` on payload. No version
bump needed because legacy cursors without `t` are gracefully accepted
(fall-through to live count); HMAC signature is computed over canonical JSON,
so old signatures keep verifying.

## G3 — Implementation
- `_build_next_cursor(arm, last_row, *, total)`: new keyword `total` threaded
  into every arm's payload as `"t"`. (None = legacy behaviour, omit.)
- Each `_search_<arm>` function:
  - Compute `total` from `COUNT(*) OVER ()`.
  - If `cursor` provided AND cursor carries `"t"` → override returned `total`
    with the cursor's snapshot value.
  - Always pass the *current* `total` (either snapshot if present, else live)
    into the next-cursor builder, so the snapshot propagates page-to-page.

## G4 — Backward compat
- Old cursors (no `t` key) decode normally; `cursor.get("t")` is `None`;
  service falls back to live `total_count` and embeds the live count as the
  snapshot into the *new* next_cursor (i.e. snapshot is established on
  whichever request is the first one after deploy).
