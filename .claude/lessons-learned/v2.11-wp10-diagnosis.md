## What was applied + where

### `app/database.py`

- New module-level constant `NAMING_CONVENTION` with the 4 keys we ship
  (`ix`, `uq`, `fk`, `pk`).
- `class Base(DeclarativeBase): metadata = MetaData(naming_convention=NAMING_CONVENTION)`
  — replaces the previous bare `class Base(DeclarativeBase): pass`. The
  override is the SQLAlchemy 2.0 idiomatic way to inject a non-default
  `MetaData` into a `DeclarativeBase` subclass.

`alembic/env.py` already wires `target_metadata = Base.metadata`, so the
convention is automatically picked up by future
`alembic revision --autogenerate` runs — any new FK / UQ / IX / PK that
doesn't get an explicit `name=` will be auto-named through the templates
instead of falling back to `None` (the WP06 footgun).

### Convention contents

```python
NAMING_CONVENTION = {
    "ix": "ix_%(column_0_label)s",
    "uq": "uq_%(table_name)s_%(column_0_name)s",
    "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
    "pk": "pk_%(table_name)s",
}
```

**`ck` is intentionally omitted.** See "Test breakage" below.

### `tests/test_naming_convention_wp10.py` (new, 3 tests)

- `test_base_metadata_has_naming_convention` — asserts the dict exactly.
- `test_unnamed_unique_constraint_gets_uq_prefix` — builds a fresh
  `MetaData` with `NAMING_CONVENTION`, attaches a `Table` with an
  unnamed `UniqueConstraint("sku")`, asserts compiled name =
  `uq_widgets_sku`.
- `test_unnamed_foreign_key_gets_fk_prefix` — same shape with
  `ForeignKey("domains.id")` on `problems.domain_id`, asserts
  `fk_problems_domain_id_domains` (matches the WP06 fix's chosen name).

### `tests/test_alembic_constraint_name_lint_wp10.py` (new, 2 tests)

- `test_no_none_named_constraint_in_alembic_versions` — walks every
  `alembic/versions/*.py`, AST-parses each, scans for `Call` nodes whose
  `func.attr` (or `func.id` for bare-import style) is in
  `{create_foreign_key, drop_constraint, create_unique_constraint,
  create_check_constraint, create_primary_key}`, flags when
  `args[0]` is `Constant(value=None)` or args is empty. **25 files
  scanned, 0 hits.**
- `test_alembic_constraint_name_lint_detects_synthetic_bad` —
  self-test covering 5 synthetic cases (bad FK, bad drop, bad UQ,
  good named pair, bare-import `create_foreign_key(None, ...)`).

## Test breakage from `naming_convention` adoption

### Symptom

After the *first* commit (which included the idiomatic `ck` template
`ck_%(table_name)s_%(constraint_name)s`), `pytest -q` showed:

```
FAILED tests/migrations/test_migration_roundtrip.py::test_each_agent_kanban_revision_is_reversible
asyncpg ProgrammingError: constraint
"ck_projects_ck_projects_coalesce_seconds_range" of relation
"projects" does not exist
[SQL: ALTER TABLE projects DROP CONSTRAINT
       ck_projects_ck_projects_coalesce_seconds_range]
```

### Root cause

The `ck` template substitutes `%(constraint_name)s` with whatever name
the caller passes — there is no "is this already prefixed" detection.
Every existing migration in this repo passes the **full** name:

```python
# alembic/versions/a18_project_coalesce_seconds.py
op.create_check_constraint(
    "ck_projects_coalesce_seconds_range", "projects", "..."
)
op.drop_constraint(
    "ck_projects_coalesce_seconds_range", "projects", type_="check"
)
```

…and many more in `a1`, `a7`, `a9`, `a10`, `a11`, `a18`. Adding the
idiomatic `ck` template wraps these to
`ck_<table>_<full-name>` (double-prefixed) at DDL compile time. Upgrade
silently created the doubled name; downgrade ran `DROP CONSTRAINT
ck_projects_ck_projects_coalesce_seconds_range` against a DB whose
constraint had been written under the same doubled name the same run,
**OR** — when fresh — the migration upgrade phase added the constraint
under a doubled name, the downgrade tried to drop that name… and asyncpg
couldn't find it because the live DB constraint was actually written
under the *non*-doubled legacy name from a previous test fixture cycle.
Either way the symptom is the same: the convention is unsafe to apply
on top of pre-existing migrations that pass full names.

### Resolution (per brief: "DO NOT alter existing constraint names in
migration files")

Dropped the `ck` key from `NAMING_CONVENTION`. The other four templates
(`fk`, `uq`, `ix`, `pk`) don't have this collision because:

- `fk` uses `%(column_0_name)s` and `%(referred_table_name)s` — no
  `constraint_name` substitution; passing an explicit `name="fk_foo"`
  doesn't hit the template at all (SQLAlchemy uses the explicit name).
- `uq` / `ix` / `pk` similarly use column-driven substitutions.
- Only `ck` uses `%(constraint_name)s`, which is the unsafe
  passthrough.

This means **future unnamed CheckConstraints get no auto-name** (their
DDL will fail to compile, surfacing the omission at dev time — also
acceptable). The lint can be tightened later to require `name=` on
every `CheckConstraint(...)` call.

### Files edited as a result

- `app/database.py` — `NAMING_CONVENTION` dict without `ck`, plus an
  inline comment explaining the trade-off.
- `tests/test_naming_convention_wp10.py` — `expected` dict matches the
  shipped 4-key convention; comment cites the omission rationale.

No production source files outside `app/database.py` changed. No
migration files were touched. No DB constraint names changed.

## Existing constraint names in code (audit)

Grepped `app/models/**.py` for `UniqueConstraint(...)` /
`CheckConstraint(...)` / `ForeignKeyConstraint(...)` declarations. All
constraints in the ORM use **explicit `name=`** strings (e.g.
`uq_solution_version_number`, `uq_agent_accounts_name`,
`ck_tickets_*`). None rely on auto-naming today. The convention is
therefore strictly forward-looking for ORM code; alembic autogenerate
is where it actually changes behavior (autogen would otherwise emit
`None`-named FKs).

## Surprises

1. **The "idiomatic" `ck` template is a footgun on legacy projects.**
   Every SQLAlchemy doc and recipe ships the same
   `ck_%(table_name)s_%(constraint_name)s`, but it only works if
   callers commit to passing SHORT names from day one. Adopting it
   mid-life requires rewriting every existing `CheckConstraint(name=)`
   to short form — which is exactly what the brief forbade. Reasonable
   pragmatic call to omit the key entirely.
2. **`fk` / `uq` / `ix` / `pk` are immune** because their templates
   substitute column/table-driven tokens, not the caller-provided
   name. Explicit `name=` arguments win cleanly. This is why WP10's
   four shipped keys work as advisory-only.
3. **Lint passes with 0 hits** — v2.10-WP06's fix was the only
   historical occurrence. Project hygiene was already good; the lint
   is forward-looking insurance.

## v2.11/v2.12 follow-ups

1. **Resurrect `ck` template via short-name port.** A future WP could
   walk every `CheckConstraint(name="ck_*_*")` in models and
   migrations, strip the redundant `ck_<table>_` prefix, and re-enable
   the idiomatic `ck` template. Non-trivial because it requires a
   migration to RENAME the live DB constraints — out of scope for
   WP10's "don't change live constraint names" charter.
2. **Tighten `CheckConstraint` lint.** Static check that every
   `CheckConstraint(...)` call (both in models and migrations) passes
   an explicit `name=`. Without the `ck` template, unnamed CKs no
   longer auto-resolve — the lint would catch the omission before
   runtime DDL fails.
3. **Wire `naming_convention` awareness into the future-migration
   review checklist.** When someone runs `alembic revision
   --autogenerate`, the generated stub should now emit named FKs
   automatically — but reviewers should still spot-check for any
   `None`-name slips, especially in `op.drop_constraint(None, ...)`
   downgrade-only paths that autogen sometimes leaves.
