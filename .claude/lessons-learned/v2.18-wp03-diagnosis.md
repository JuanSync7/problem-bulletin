## Backend payload shape (source of truth)
`app/services/leaderboard.py` emits **two** row shapes, switched on the request
`track` parameter. Both share `user_id`, `display_name`, `rank`.

| Track       | Score column     | Row dict keys                                     |
|-------------|------------------|---------------------------------------------------|
| `solvers`   | `accepted_count` | `user_id`, `display_name`, `accepted_count`, `rank` |
| `reporters` | `upstar_count`   | `user_id`, `display_name`, `upstar_count`, `rank`   |

Wrapped in `LeaderboardResponse { track, period, entries: list[LeaderboardEntry] }`
where `LeaderboardEntry` carries `model_config = ConfigDict(extra="allow")` to
let the track-specific score key flow through.

No per-row discriminator field is emitted by the backend, so the natural
discriminator is the **request track**, not a runtime tag.

## Union shape landed
```ts
interface LeaderboardEntryBase {
  user_id?: string; userId?: string;
  display_name?: string; displayName?: string;
  rank?: number;
}
interface LeaderboardSolverRaw   extends LeaderboardEntryBase { accepted_count?: number; }
interface LeaderboardReporterRaw extends LeaderboardEntryBase { upstar_count?: number;   }
type     LeaderboardRawEntry  = LeaderboardSolverRaw | LeaderboardReporterRaw;
```

All score-source fields are optional ‚Äî defensively tolerates missing keys
without falling back to `any`. camelCase aliases retained for any legacy
in-flight payloads (pre-WP02 callers).

## Normaliser
```ts
normalizeLeaderboardEntry(raw, track, index) ‚Üí LeaderboardEntry
```
- Switch on the `track` argument (NOT runtime sniffing).
- `solvers` ‚Üí `score = raw.accepted_count ?? 0`
- `reporters` ‚Üí `score = raw.upstar_count ?? 0`
- `default` branch holds a `const _exhaustive: never = track` to fail compile
  if a new track is added without a normaliser arm.
- Used in `fetchLeaderboard` via `raw.map((e, i) => normalizeLeaderboardEntry(e, t, i))`.

The previous ad-hoc fallback `e.score ?? e.accepted_count ?? e.upstar_count ?? e.problem_count`
included a `problem_count` slot, but `problem_count` is **not** emitted by the
current backend service. Dropped ‚Äî no production callsite exercises it.

## Per-track surprises / follow-ups
1. Backend `LeaderboardEntry` schema uses `extra="allow"` to ferry the
   track-specific score column ‚Äî schema does not formally type the variant.
   Could be tightened backend-side later (split into `SolverEntry` /
   `ReporterEntry` Pydantic models + `Union` response). Out of scope for L3.
2. `problem_count` was a phantom fallback in the original normaliser
   (no backend emits it). Removed cleanly ‚Äî if a future track introduces
   it, the exhaustiveness `never` will force the design choice front-and-
   centre instead of silently working.
3. The frontend `Track` literal type is now `export`ed for the test file.
   No other module imports it; benign widening of the public surface.
