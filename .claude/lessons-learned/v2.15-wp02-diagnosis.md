# v2.15-WP02 (C2) — bare `catch {}` structural lint — diagnosis

## Context

v2.14-WP04 retrospective surfaced the lesson **"`catch {}` is structurally
invisible to the polarity audit"**. WP04 had to promote every bare
`catch {}` in admin pages to `catch (err)` ad-hoc, then thread
`err.message` to the UI. A structural lint over `frontend/src/pages/**`
would have surfaced these mechanically and prevents regression.

WP02 is the **PIN**: enumerate today's offenders into an explicit
allow-list, fail loud on any NEW offender, fail loud on any STALE entry.
WP03 is the **SWEEP**: it deletes allow-list entries as it migrates
category-c silent-on-non-2xx page sites.

## Files added

- `frontend/src/pages/__tests__/catch_block_lint.test.ts` — single file
  containing the lint helper (`scanSource`), the offender allow-list
  (`_OFFENDER_ALLOWLIST`), and 7 vitest cases. No helper module split:
  the entire lint is ~250 LOC and lives in one file for readability,
  mirroring how `tests/test_create_app_factory_lint_wp09.py` co-locates
  the scanner with its tests.

## Implementation choices

- **TS compiler API.** Uses `ts.createSourceFile` + `ts.forEachChild` to
  walk and match `CatchClause` nodes. The `typescript` package was
  already a dev dep (`5.5.3`); no new dep added. Custom regex over JSX
  was explicitly rejected — too fragile around nested string literals,
  template strings, and JSX expression containers.
- **Two offender kinds.**
  - `bare` — `CatchClause.variableDeclaration` absent (`catch { ... }`).
  - `swallow` — `CatchClause.variableDeclaration` present with one or
    more binding names, but the block has zero `ts.Identifier` references
    matching any binding name. A rethrow (`catch (err) { throw err; }`)
    references `err`, so it is NOT a swallow.
- **Edge cases handled.**
  - `catch (err: unknown)` typed bindings — the type annotation is on
    `variableDeclaration.type`, not on the binding name. The walker reads
    the identifier name directly, so typed bindings work.
  - `try/catch/finally` — only the `catchClause` is examined; the
    `finallyBlock` is irrelevant.
  - Nested catches — `ts.forEachChild` recurses; nested `CatchClause`
    nodes are visited.
  - Object/array destructuring bindings — `collectBindingNames` walks
    `ObjectBindingPattern`/`ArrayBindingPattern` and collects every bound
    `Identifier`. The lint succeeds if ANY of the bound names is
    referenced in the block.
- **`_OFFENDER_ALLOWLIST` shape.** `Array<{file, line, kind, justification}>`
  rather than a flat `Set<"file:line">`. The justification field is
  inert (the test does not read it) but documents intent for WP03 —
  every entry should describe whether WP03 needs to surface the error
  to UI or whether the silent path is genuinely idiomatic (e.g.
  date-parse fallback, localStorage best-effort).

## Self-tests (the synthetic-bad / synthetic-good battery)

1. **Bare detected.** `try {} catch { /* swallow */ }` → 1 hit, kind=`bare`.
2. **Swallow detected.** `try {} catch (err) { setError('oops'); }` → 1
   hit, kind=`swallow`. The body has no `err` reference; the catch
   silently drops the structured error.
3. **Consumed binding NOT flagged.** `catch (err) { console.error(err); }`
   → 0 hits. `err` is referenced.
4. **Rethrow NOT flagged.** `catch (err) { throw err; }` → 0 hits.
   Rethrow consumes the binding.
5. **Typed binding NOT flagged.**
   `catch (err: unknown) { setError((err as Error).message); }` → 0 hits.
   The `unknown` annotation is not in the way of the identifier walk.
6. **Nested / finally.** A nested `catch {}` inside an outer
   `catch (e) { setError(e); } finally { cleanup(); }` flags exactly
   the inner bare catch and ignores the outer (consumed) and `finally`.
7. **Allow-list lockstep (the main lint test).** The 36 detected
   offenders MUST equal the 36 allow-list entries. Any new offender
   (a future PR re-introducing the class) fails loud at `file:line`;
   any stale entry (WP03 fixed a site but forgot to remove from the
   allow-list) also fails loud.

## Out-of-scope (WP03 is the SWEEP)

WP02 did NOT migrate any offender. Each allow-list entry's
`justification` records the rough rationale; WP03's job is to:

1. Decide UX per site (toast / inline error / page-level error /
   genuinely idiomatic silent fallback).
2. Promote `catch {}` → `catch (err) { ... }` or restructure the
   `if (res.ok)` block where appropriate.
3. **Delete the allow-list entry** in the same commit that fixes the
   site. The lint will fail loud on a stale entry, forcing the cleanup.

## Lessons captured (for v2.15 retrospective)

1. **Structural lints over TS source pay back like Python AST lints
   do.** The pattern from `tests/test_create_app_factory_lint_wp09.py`
   ports almost line-for-line to TS — the only differences are the AST
   library (`typescript` vs `ast`), the file extension scan, and the
   test framework (vitest vs pytest). The `_ALLOWLIST` discipline
   (entries documented, stale detection, justification-required) is
   what makes the lint a forcing function rather than dead weight.
2. **JSDoc block comments break on `**/*` glob fragments.** The first
   draft's header comment said "scans `frontend/src/pages/**/*.tsx`",
   which esbuild parses as "end of comment after `*/`, then start of
   regex". The fix is prose — "tsx files under `frontend/src/pages/`"
   — rather than a glob in a comment. Cheap lesson, cheap fix.
3. **Promote-to-`catch (err)` already happened; the residual class is
   bare-catch.** Zero swallow offenders today means v2.14-WP04's
   adapter sweep finished the bound-but-discarded class. The remaining
   work is the category-c sites where the page has no error UX at all.
   That maps exactly to WP03's scope.
