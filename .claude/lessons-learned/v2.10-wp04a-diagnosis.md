# v2.10-WP04a — Core problem lifecycle live-DB port

## TL;DR

**91 deferred IDs ported to live-Postgres**, sourced from four service-layer
test files (`test_voting.py` 12, `test_solutions.py` 22, `test_comments.py`
23 [the 5 boundary IDs of `test_problems.py` ride along], `test_problems.py`
34 = 91 net). All 91 IDs are bucket **(c) — tests assert the wrong
invariant**: every file was written against an imagined service contract
(dict returns, alternate kwargs like `parent_type`/`payload`/`current_user`,
mocked `mock_db` reads) that the production code has never matched. The
remediation is therefore a test-side rewrite onto the WP02 live-`db`
session fixture and the *actual* service signatures.

No production source under `app/` was edited. Full suite final:
`1026 passed / 0 failed / 168 skipped / 14 xfailed` (was 935 / 259 / 14).

deferred ID was written against a contract that production never honoured.

The natural follow-up question — *why* were these tests so divergent? —
echoes WP01's `test_v1_routes_gone.py` finding: the v1→v2 migration shipped
a *parallel* service surface, and the old test files were never
re-pointed at the live shape. The mock-DB infrastructure papered over the
divergence because nothing exercised the real signature.

## Production fixes (G2)

**None.** Every failure was bucket (c). The diagnosis ran on each file:

1. Read the service module (`app/services/{voting,solutions,comments,problems,feed}.py`).
2. Read the file's caller in `app/routes/` to confirm the production contract.
3. Compared with the test file's assertion shape and call signature.
4. Where they diverged, the route layer always matched the service —
   never the test. Conclusion: test is wrong, service is right.

Per the WP brief: when service has drifted significantly from test intent
the test is updated, not the service. That rule applied to all 91 IDs.

## Schema drift surfaced

The `problems` table is mapped as ORM column `Problem.status` →
*database* column `legacy_status` (renamed in the `a1_agent_kanban`
migration). Seed helpers must INSERT into `legacy_status`, not `status`.
The first iteration of `tests/helpers/seed_problem.py` failed loudly
against `UndefinedColumnError: column "status" of relation "problems"
does not exist`, which surfaced this immediately. Captured in the helper
as a single-source insert; no production change needed because the ORM
column-mapping already abstracts the rename.

This is worth a v2.11 follow-up: the dual name (`status` in code,
`legacy_status` in DB) is a footgun for every new raw-SQL test helper.
Either rename the DB column back, or write a project-level lint that
flags raw `INSERT INTO problems ... status ...` outside the migration
files.

## v2.11 follow-ups discovered

1. **Rename `problems.legacy_status` back to `status`, or document
   prominently.** Every new test helper that touches the table risks the
   same `UndefinedColumnError` we hit on the first iteration. A
   `tests/helpers/seed_problem.py` docstring is a poor substitute for a
   schema lint. Candidate v2.11 migration: rename column + sweep raw
   SQL.

2. **`AGENT_BULLETIN_TEST_DOCS.md` references in test docstrings now
   point to a phantom contract.** Every WP04a test file's module
   docstring says "Derived from docs/AION_BULLETIN_TEST_DOCS.md — …
   section". That document either documented a never-shipped API or
   pre-dates the route-layer refactor. A v2.11 doc-sweep should either
   reconcile the spec or delete the references; otherwise future
   debuggers will follow a dead trail.

3. **`pin_problem` does not enforce admin role at the service layer.**
   The service accepts `admin_id` for audit, but never checks
   `User.role == admin`. The route layer enforces it. The test file
   pins the *service contract* (admin_id is accepted as-is); a v2.11
   could either tighten the service or rename the parameter to
   `actor_id` to remove the implication.

4. **Cursor encoding helpers belong in a public API surface.** The
   `test_pinned_not_on_subsequent_pages` test had to import
   `encode_cursor` from `app.services.feed` to build a forward
   cursor. Pinned-suppression is a contract — having tests reach into
   the service for cursor construction is fragile. Consider exposing
   a `Cursor` value type with a public `encode/decode` API.

5. **Anonymous masking is split between service and route layers.**
   `app/services/solutions.py:_solution_to_dict` carries a comment
   noting that admin reveal is a route-layer concern but does not
   enforce it. WP04a's `test_author_revealed_to_admin` had to
   accommodate that split. A v2.11 could consolidate to one layer —
   most likely the service, with the route passing the viewer's role
   in — so the test can pin one invariant per location.

## Surprises

- The 91 IDs looked like 4 unrelated clusters, but every one had the
  same shape: tests written from a spec doc that never matched
  production. The diagnosis collapsed to a single rule ("update the
  test"), which kept the WP small.
- WP01's manifest classified these as "bucket (a) — mock-DB rot". The
  bucket was wrong in the same way WP02's "Bucket D" was wrong:
  mock-DB rot was real, but the tests *also* asserted a contract that
  never existed. Bucket (c) — wrong invariant — is the more accurate
  attribution. Restated for WP04b/04c: when porting a mock-DB test,
  always sanity-check the call signature against the live service
  before assuming the mock was the only thing wrong.
- The 5 `TestBoundaryConditions` IDs in `test_problems.py` are pure
  Pydantic schema tests — they touch no DB at all. They were deferred
  as collateral when the rest of the file went red. Confirmed by
  running them directly: they pass without the conftest skip-hook,
  showing they had no test-side rot. Worth a v2.11 lint: a deferred
  test that has no fixture dependency on the broken machinery probably
  doesn't need to be deferred at all.
- The first iteration of `seed_problem` hit a schema-drift wall
  (`legacy_status`). That cost ~5 minutes to diagnose and produced a
  permanent fix in the helper — small, well-contained, captured in
  the follow-ups above.
- Solution `is_anonymous` admin reveal is a route-layer concern that
  the service docs flag explicitly but the test files never noticed.
  The new `test_author_revealed_to_admin` pins the service-level
  contract (anonymous flag preserved) rather than the route policy.
