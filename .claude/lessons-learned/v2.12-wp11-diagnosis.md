# v2.12-WP11 (Bucket C3) — Diagnosis

## Framing

WP11 adds a CI lint that compares every in-scope FastAPI response schema
(via `app.openapi()`) against the matching TypeScript response interface
in `frontend/src/api/*.ts`. The lint catches field-level drift that
would otherwise only surface at runtime (e.g. the v2.11-WP07
`TicketRead` narrowing that WP06 introspection accidentally pinned).

The test file landed in a half-completed state from a previous agent:
6 passing, 6 failing. Three failures were in the synthetic parser
self-tests; three were in the parametrised real-route checks (two
genuine drifts plus one parser-found-nothing case).

## Parser bug: greedy `[^{]*` consumed across lines

The parser regex was:

```python
_TS_BLOCK_RE = re.compile(
    r"^export\s+(?:interface|type)\s+(?P<name>\w+)\b[^{]*\{",
    re.MULTILINE,
)
```

Two compounding problems:

1. **Self-test fixtures are indented.** The synthetic-bad strings in
   `test_parser_self_test_*` are defined inside Python triple-quoted
   blocks with 4-space indentation, so `^export` (start-of-line) never
   matches even with `re.MULTILINE`. All three self-tests blew up with
   `TsParseError: FakeResp: not found in source`.
2. **Union-typed aliases swallow the next interface.** `sprints.ts`
   begins with
   `export type SprintState = "planned" | "active" | "closed";` followed
   by `export interface SprintDTO { ... }`. `[^{]*` matches across
   newlines (only `.` excludes newlines), so the first match started at
   `export type SprintState` and consumed all the way to the `{` of
   `SprintDTO`. `finditer` doesn't re-scan that span, so `SprintDTO`
   was effectively invisible — `parse_ts_type("SprintDTO")` raised
   `not found in source`. The TS `SprintDTO` shape is actually a
   superset of the OpenAPI shape, so this was a false-negative caused
   by the parser, not a real drift.

### Fix (chose: fix the parser, not the fixtures)

Single-line regex change:

```python
r"^\s*export\s+(?:interface|type)\s+(?P<name>\w+)\b[^{;]*\{"
```

- `^\s*export` tolerates indented synthetic test fixtures.
- `[^{;]*` stops at `;`, so a union/enum-style `export type X = "a" | "b";`
  declaration terminates at its own line and the next `export interface`
  is matched independently.

Both real `.ts` files declare types at column 0, so `\s*` doesn't
loosen the parser against shipped code. Fixing the parser (vs. the
fixtures) is the conservative choice — the parser is the production
artifact and the fixtures exist to *exercise* it; a parser that can't
handle indented input or co-located type aliases is the actual bug.
