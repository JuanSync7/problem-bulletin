## Problem framing

v2.10 left a Python-attribute-vs-DB-column asymmetry: ORM exposed
`Problem.status` but DB column was `legacy_status`. v2.10-WP04b fixed one
raw-SQL hit, v2.11-WP02 swept the rest and pinned a regression lint. The
asymmetry persisted as a footgun. Bucket E2 of the v2.11 plan closes it by
renaming the DB column back to `status` now that the work-tracker `tickets`
table is physically separate (split off in `a8_finalize_ticket_split`) and
the original collision is gone.

## Iteration log

Single iteration. One hiccup:

- First revision string `a19_rename_problems_legacy_status_to_status`
  (43 chars) overflowed `alembic_version.version_num VARCHAR(32)` —
  `StringDataRightTruncationError` on upgrade. Renamed the revision ID to
  `a19_problems_status_rename` (26 chars) and updated cross-references in
  `app/models/problem.py`, `app/services/search.py`,
  `tests/test_legacy_status_sweep.py`, and `tests/services/test_search.py`.
  File path kept the longer descriptive name for grep-ability.

Lesson pin: alembic's default `alembic_version` table column is
VARCHAR(32). Revision strings longer than 32 chars upgrade locally
(SQLite is permissive) but fail under Postgres-asyncpg at the final
`UPDATE alembic_version SET version_num=...` step. Keep revision IDs
≤ 32 chars; descriptive file names can be longer.
