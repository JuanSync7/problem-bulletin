## Unpinned-type inventory (audit of `frontend/src/api/*.ts`)

Sourced via:

```
grep -n 'Page<\|CursorPage<\|interface.*<\|type.*<' frontend/src/api/*.ts
grep -n '^export type.*=' frontend/src/api/*.ts
```

### Generic wrappers (`Page<T>`-shaped)

| Backend route | OpenAPI schema | TS file | TS expression | Inner schema pair |
|---|---|---|---|---|
| `GET /api/v1/projects` | `Page_ProjectRead_` | `projects.ts` | `Page<ProjectDTO>` | `ProjectRead ↔ ProjectDTO` (already pinned by WP11 inner) |
| `GET /api/v1/projects/{id}/members` | `Page_ProjectMemberRead_` | `projects.ts` | `Page<ProjectMemberDTO>` | `ProjectMemberRead ↔ ProjectMemberDTO` (pinned by WP11) |
| `GET /api/v1/projects/{id}/components` | `Page_ComponentRead_` | `projects.ts` | `Page<ComponentDTO>` | `ComponentRead ↔ ComponentDTO` (pinned by WP11) |
| `GET /api/v1/sprints` | `Page_SprintRead_` | `sprints.ts` | `Page<SprintDTO>` | `SprintRead ↔ SprintDTO` (pinned by WP11) |
| `GET /api/v1/notifications` | `Page_TicketNotificationRead_` | `notifications.ts` | `Page<TicketNotification>` | `TicketNotificationRead ↔ TicketNotification` (pinned by WP11) |

The wrapper itself is a 3-field flat envelope: `items`, `next_cursor`,
`total` — both sides emit all three. `total` is nullable on both sides
(snapshot-total mode, v2.11-WP14). WP05 pins the WRAPPER field-set so a
backend regression that drops `total` (or renames `next_cursor` to
`cursor`) fails CI.

Routes that return `Page_*` but whose TS does NOT use `Page<T>` (instead
inlines the response shape):

| Route | OpenAPI schema | TS handling |
|---|---|---|
| `GET /api/v1/tickets/search` | `Page_TicketRead_` | Inline `{items: TicketDTO[]; next_cursor?; total?; column_counts?}` consumed via `TicketsPage extends Page<TicketDTO>`. **TicketRead is `extra=allow` ⇒ permissive; same skip rationale as WP11 inner.** |
| `GET /api/v1/tickets/{id}/watchers` | `Page_TicketWatcherRead_` | No hand-written TS type — UI consumes inline. SKIP (no TS counterpart). |
| `GET /api/v1/tickets/{id}/attachments` | `Page_TicketAttachmentRead_` | Same — no hand-written TS type. SKIP. |
| `GET /api/agents/activity`, `GET /api/v1/agents/activity` | `Page_AgentActivityItem_` | UI inlines the type. No hand-written TS wrapper. SKIP (no TS counterpart). |
| `GET /api/problems` | `CursorPage_ProblemResponse_` | Bulletin domain; no hand-written `CursorPage<T>` in `frontend/src/api/*.ts`. SKIP (out of scope for the ticketing api dir). |

### Discriminated unions

| Backend schema | TS type | Branch count | Discriminator |
|---|---|---|---|
| `ActivityPage.items[*]` ⇒ `oneOf [TransitionRead, CommentRead, LinkRead]` with `discriminator.propertyName=kind` | `ActivityItem = TransitionActivityItem \| CommentActivityItem \| LinkActivityItem` | 3 ↔ 3 | `kind` ("transition" \| "comment" \| "link") |
| `SearchV2Response` (5 optional `SearchArm \| null` arms) | `SearchV2Response` (5 optional `SearchArm \| null` arms) | flat — handled by WP11 parser, but it currently SKIPS because the WP11 prompt didn't pin it. WP05 adds the inner `SearchArm ↔ SearchArm` pair. | n/a |
| `SearchArm` ↔ `SearchArm` | flat closed schema with `items: SearchItem[]`, `total`, `next_cursor`, `total_authority` | 4 ↔ 4 fields | n/a (items[] recurse target is `SearchItem` which is permissive — skip recursion per WP11 polarity rule) |

### Permissive / out-of-scope (continued skip — documented decisions)

- `TicketRead ↔ TicketDTO`, `SearchItem ↔ SearchItem` — both `extra=allow` AND TS has `[k: string]: unknown`. Field-set parity is a tautology.
- `ActivityPage` outer wrapper — `oneOf items` recursion is already covered by the discriminated-union pair below; the outer field-set (`items`, `next_cursor`, `total`) is identical to `Page<ActivityItem>` shape so it's also covered by the generic pair test for ActivityPage.
- `CursorPage_ProblemResponse_` — bulletin domain. No `CursorPage<T>` in `frontend/src/api/*.ts`. Deferred to a future bulletin-domain WP.
- Intersection types (`A & B`), mapped types (`{ [K in T]: ... }`), conditional types — TypeScript syntax this codebase doesn't use today; out of WP05 scope.

## Parser syntax supported in WP05

The WP05 helper module (`tests/_wp05_ts_parser.py`) imports the WP11
flat-interface parser and adds two narrow capabilities:

1. **Generic interface**:
   ```ts
   export interface Page<T> {
     items: T[];
     next_cursor: string | null;
     total: number | null;
   }
   export interface TicketsPage extends Page<TicketDTO> { column_counts?: ... }
   ```
   The parser reads the wrapper definition (with `<T>` stripped), captures
   the property names, and records which property is the generic-
   parameterised array. `parse_ts_generic_wrapper("Page", source)` returns
   `{"properties": {"items","next_cursor","total"}, "generic_item_prop":
   "items"}`. The consumer call `Page<Foo>` is resolved by name match —
   `Foo` is then handed off to the existing WP11 flat parser for inner
   parity if the caller asks.

2. **Discriminated union alias**:
   ```ts
   export type ActivityItem =
     | TransitionActivityItem
     | CommentActivityItem
     | LinkActivityItem;
   ```
   `parse_ts_union_alias("ActivityItem", source)` returns
   `["TransitionActivityItem", "CommentActivityItem", "LinkActivityItem"]`.
   Branch ordering is preserved so error messages line up with OpenAPI
   `oneOf` order.

3. **Inline `T[]` array property** — `items: TicketRead[]` is parsed as
   property `items` with element type `TicketRead`. Used to support the
   generic-wrapper resolution.

### Explicitly NOT supported (raises `TsParseError` → caller MUST document the skip)

- Generic type *aliases* (`type Page<T> = { ... }`). Not used in this
  codebase; would require additional parser branches.
- Intersection types (`A & B`).
- Mapped/conditional types.
- Nested generics in the param (`Page<Map<string, T>>` etc.).
- Multi-parameter generics (`<T, U>`).
- Default generic parameters (`<T = unknown>`).

If any of these shapes appears in a future PR, the parser raises and the
caller must add either a parser branch or a documented skip.

### Regression note: the WP11 `finditer` greedy-block bug

v2.12-WP11 had a regex `[^{]*\{` that could (in theory) consume across
newlines and swallow a `type X = A | B;` alias into the next interface's
brace block. WP05 inherits the WP11 fix (`[^{;]*\{`) and adds an
explicit regression self-test (`test_wp05_parser_does_not_swallow_union_into_next_interface`)
that fails if the regex is loosened again.
