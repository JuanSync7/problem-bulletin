## G2 тАФ Route handlers returning raw structured dicts

Total raw-dict returns enumerated: **36** across `app/routes/`.

**Trivial (do not migrate)** тАФ `{"status": "ok"}`, `{"detail": "..."}`, etc.:
- `app/routes/health.py:30,32,34,54,56,58` тАФ health probes
- `app/routes/edit_suggestions.py:144,174` тАФ `{"status": "accepted/rejected", "id": ...}`
- `app/routes/problems.py:206` тАФ `{"detail": "Claim removed", "claimed": False}`
- `app/routes/voting.py:28,39` тАФ `{"active": bool, "count": int}` (small & stable)
- `app/routes/search.py:209,210` тАФ `{"results": [...]}` thin wrapper

**Structured payloads тАФ FLAGGED for v2.11 (G6 deferred):**
1. `app/routes/edit_suggestions.py:67` тАФ `create_edit_suggestion` returns ad-hoc dict instead of `EditSuggestionResponse` (the response model is defined but not used as `response_model=` and not returned). Low-risk migration, but ticket-it.
2. `app/routes/comments.py:258` тАФ comment payload
3. `app/routes/leaderboard.py:42` тАФ leaderboard payload
4. `app/routes/users.py:41, 75` тАФ user payload (already partly via `UserResponse` legacy schema тАФ needs consolidation)
5. `app/routes/problems.py:216` тАФ problem detail payload
6. `app/routes/tickets.py:86,372,565,584,620,652,676,685` тАФ large surface; ticket views often hand-built. Migration risk is non-trivial because shapes are negotiated with the frontend; defer to v2.11.
7. `app/routes/agents.py:97`, `app/routes/projects.py:83,154,228`, `app/routes/sprints.py:51`, `app/routes/tickets.py:249,707,775` тАФ paged `{"items": [m.to_dict() for m in rows], ...}` shapes. Should converge on `Page[T]` (already defined in `app/schemas/common.py`). Mechanical but cross-cuts many routes/tests тАФ defer.

**Rationale for deferring:** these dict shapes are stable, type-checked at the route boundary by the frontend, and migrating them touches ~15 routes + many test snapshots. The user's directive is "make sure everything is pydantic"; the *models* all are. The route serialization style is a separate convergence project.

## G3 тАФ Service-layer non-Pydantic types

Single finding:
- `app/services/context.py:17` тАФ `@dataclass(frozen=True) class Actor`. **Intentional тАФ not a DTO.** This is a contextvar-bound principal carrier (per WP impl ┬з2.3). Frozen dataclass is appropriate: it's instantiated once per request by auth middleware, never serialized to/from JSON, never crosses a route/service API boundary as payload. **Keep as dataclass.**

No `TypedDict` or `NamedTuple` in `app/services/`.

## G8 тАФ Deferred to v2.11 (backlog)

Tracked items for a future "route serialization convergence" work-package:

1. **WP тАФ Edit-suggestion route uses its response model.** `app/routes/edit_suggestions.py` defines `EditSuggestionResponse` but `create_edit_suggestion` returns a hand-rolled dict. Wire `response_model=EditSuggestionResponse` and return the model.
2. **WP тАФ Converge paged list routes on `Page[T]`.** `agents.py`, `projects.py`, `sprints.py`, `tickets.py` paged list handlers return ad-hoc `{"items": [...], "limit": ..., "offset": ...}` instead of `app.schemas.common.Page[T]`. Mechanical but cross-cuts ~15 routes and snapshot tests.
3. **WP тАФ Ticket detail/comments/leaderboard/users/problems response models.** Define Pydantic response models for the larger structured payloads in `app/routes/{tickets,comments,leaderboard,users,problems}.py`. Frontend contract negotiation required.
4. **(Out of scope, but noted)** `app/services/context.py:Actor` stays as `@dataclass` тАФ context-var principal carrier, not a DTO. No action needed.

## Surprises

- The codebase is in **very** good Pydantic-v2 shape. Only **one** v1 idiom existed across all of `app/`, and it was in an inline route schema, not the canonical `app/schemas/` package тАФ which is 100% clean v2.
- Zero `.dict()` / `.parse_obj()` / `from_orm()` / etc. callsites тАФ suggests prior migration work (likely a v2.0/v2.1-era sweep) was thorough.
- The largest gap is *not* Pydantic-idiom тАФ it's route-handler serialization style (raw dicts vs response models). That's a UX/contract project, not a Pydantic-cleanliness project.
