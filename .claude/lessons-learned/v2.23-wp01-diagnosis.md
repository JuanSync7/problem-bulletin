### WP02 recommended target

**N = 2 candidates** (below the 3-8 target → downgrade to "small expansion + declare floor"):

1. **`GET /api/v1/tickets/search` → `Page_TicketRead_` ↔ `TicketsPage`/`Page<TicketDTO>`**
   (WP05_PAGE_PAIRS entry — clean Page<T> generic pin; TicketsPage extends Page<TicketDTO>,
   the generic-item-prop assertion exercises both `items: T[]` and the inner ref → TicketDTO).
2. **`GET /api/v1/audit-log` → `AuditLogPage` ↔ `AuditLogPage`** (auditLog.ts)

The other three unpinned Page schemas (`Page_AgentActivityItem_`,
`Page_TicketWatcherRead_`, `Page_TicketAttachmentRead_`, `CursorPage_ProblemResponse_`)
are NOT pinnable in WP02 because they have no hand-written TS Page<T>
counterpart. They become candidates only AFTER Option (d) introduces
the corresponding TS interface (Watcher case) — and even then, the
*Page-wrapper* pin is only worth it if the frontend chooses
`Page<TicketWatcher>` over a flat `TicketWatcher[]`.

## Option (d) recon — TicketWatcherRead vs TicketAttachmentRead

### Schema sizes

**`TicketWatcherRead`** (`app/schemas/tickets.py:208-215`) — **5 fields**, `extra="allow"`:
```python
id: UUID
ticket_id: UUID
watcher_id: UUID
watcher_type: Literal["user", "agent"]
created_at: datetime
```

**`TicketAttachmentRead`** (`app/schemas/tickets.py:218-230`) — **10 fields**, `extra="allow"`:
```python
id: UUID
ticket_id: UUID
uploaded_by: UUID
uploaded_by_type: Literal["user", "agent"]
filename: str
content_type: str
byte_size: int
storage_path: str
agent_step_id: str | None = None
created_at: datetime
```

### Frontend usage

Both routes have **zero frontend callers today**:
* `grep -rn '/watchers\|/attachments\|TicketWatcher\|TicketAttachment' frontend/src/` returns
  only problem-attachments (different resource: `/api/problems/{id}/attachments` —
  unrelated upload UX in `Submit.tsx` / `ProblemDetail.tsx`), and an Activity-feed
  CSS class for the `ticket_watcher_added` notification badge.
* `frontend/src/pages/TicketDetail/index.tsx` does NOT render watchers or attachments.

This means v2.23-WP03 is *not* a strict any-tightening — there is no
`any` to tighten. It is a **net-new typed-consumer introduction**.
The v2.22 seed wording ("frontend currently uses `any` or untyped JSON")
is technically accurate: the frontend uses **no JSON at all** for these
routes. Pinning them adds a fresh contract surface.

### Recommendation: **TicketWatcherRead**

Rationale (smaller, more contained blast radius):
* **5 fields vs 10** — half the schema surface to pin.
* **No binary/file metadata** — TicketAttachmentRead has `byte_size`,
  `content_type`, `storage_path` which would invite questions about
  download URLs / signed-URL strategy / agent-upload UX. Out of scope
  for a parity-lint expansion WP.
* **Discriminator already in schema** — `watcher_type: Literal["user","agent"]`
  mirrors the assignee-type pattern landed in WP47-WP49; frontend
  already has `assignee_type` chip rendering to crib from. No new UX
  primitive needed.
* **`extra="allow"` is identical on both** — TS side will use closed
  field set; lint's permissive-side polarity (refines rule from WP11)
  will treat OpenAPI as the authority. Same polarity as
  TicketAttachmentRead, so this isn't a differentiator.

### Blast-radius estimate (TicketWatcherRead)

| Surface                                   | Estimate                                   |
| ----------------------------------------- | ------------------------------------------ |
| New TS interface (tickets.ts)             | ~12 lines (interface + JSDoc)              |
| New caller hook (tickets.ts)              | ~20 lines (`listWatchers(idOrKey, params)` returning `Promise<Page<TicketWatcher>>`) |
| Existing pages to migrate                 | 0 (no existing callers)                    |
| WP11 lint entry                           | +1 tuple in `WP11_ROUTES` (TicketWatcherRead ↔ TicketWatcher) |
| Optional WP02 Page<T> pin                 | +1 tuple in `WP05_PAGE_PAIRS` if interface uses `Page<TicketWatcher>` |
| ts-any-lint allow-list entries cleared    | 0 (no existing `any` sites for watchers)   |
| Test coverage                             | New `ticketWatcherDto.test.ts` mirroring `ticketDto.test.ts` (~15 lines) |

Total new code: **~50 lines frontend + 1-2 lint tuples**. No page-level
edits, no existing-call-site migration. Net-new surface, zero migration
risk.

### Deferred to a future version

* **TicketAttachmentRead** — defer until the frontend introduces an
  attachments UX (file picker, download URL strategy, agent-step
  attachment surfacing). At that point pin attachments + the
  `byte_size` / `content_type` / `agent_step_id` field set together.
* **CursorPage_ProblemResponse_** — defer until the problems API gains
  a hand-written TS Page wrapper. Currently uses inline pagination
  shape.
* **Page_AgentActivityItem_** wrapper-shape pin — defer until audit.ts
  upgrades from `ActivityEntry[]` to `Page<ActivityEntry>` (the
  endpoint already returns a Page envelope; the frontend strips it).
  Forward-looking refactor, not a parity expansion.

## WP02 vs WP03 ordering recommendation

**WP02 (Option c) first, then WP03 (Option d), then WP04 closure.**

One-line justification: WP02 is a 2-tuple lint-append against existing
TS code (zero source changes), making it the smaller-safer-first WP;
WP03 introduces new frontend source, so landing the parity floor first
means WP03's pin is added in the SAME shape WP02 just blessed.

Sub-points:
* WP02 = `WP05_PAGE_PAIRS += (search, Page_TicketRead_, …)` and
  `WP05_FLAT_PAIRS += (audit-log, AuditLogPage, …)`. No app/frontend
  source modified. First-run-clean expected per rule (rr) — but PIN
  deterrent value is what we're buying.
* WP03 = new `TicketWatcher` interface + `listWatchers` in tickets.ts +
  WP11 tuple + optional WP05 wrapper tuple. Net-new code, one new test
  file. Rule (qq): re-count tickets.ts line offsets at WP03 start; the
  Page<T> decl currently lives at line 211, callers at 277/316/501 —
  expect ±5-line drift if any v2.23 prior-WP work shifts it.
