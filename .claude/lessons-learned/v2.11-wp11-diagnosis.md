## C8 — SPA catch-all hardening

### Current mechanism (verified in `app/main.py`)

```python
frontend_dist = Path(__file__).resolve().parent.parent / "frontend" / "dist"
if frontend_dist.is_dir():
    app.mount("/assets", StaticFiles(directory=...), name="static")

    @app.get("/{full_path:path}")
    async def serve_spa(full_path: str):
        ...
```

- Conditional on `frontend/dist` existing (not on at test time unless built).
- Implemented as `@app.get` decorator (a `Route`, NOT a `Mount`).
- Appended LAST — after every `app.include_router(...)` and every
  `app.add_exception_handler(...)`.
- Pattern: `/{full_path:path}`.

### Helper choice: option (b)

`tests/helpers/test_routes.py::register_test_route(app, path, endpoint, *, methods=("GET",), name=None) -> None`

- Calls `app.add_api_route(...)` to build a proper `APIRoute` with FastAPI's
  DI wiring.
- Pops the just-appended route off the tail of `app.router.routes` and
  re-inserts at `[0]`.

Smaller patch than (a) (sub-router refactor would touch production wiring);
gives tests an explicit "win against catch-all" knob.

### Regression-shape pin

`tests/test_spa_catchall_hardening_wp11.py::test_spa_catchall_shape_in_create_app`
inspects `inspect.getsource(create_app)` and asserts:

1. `@app.get("/{full_path:path}")` decorator still present (NOT converted
   to a Mount).
2. Still guarded by `frontend_dist.is_dir()`.
3. Catch-all line index > both `app.include_router` and
   `app.add_exception_handler` line indexes (i.e. registered LAST).

Plus a runtime cross-check that runs only when `frontend/dist` exists: the
live app has exactly one `Route` at `/{full_path:path}`.

Source-text inspection is appropriate here because the load-bearing fact
is **registration order inside `create_app`** — a runtime route-list check
wouldn't catch a refactor that, say, moves the catch-all earlier in the
function but ships before some new router include.

### Helper smoke tests

- `test_register_test_route_wins_against_catchall` — builds a fake-catchall
  app, registers a test route, asserts the test route wins AND the
  catch-all still serves unrelated paths.
- `test_register_test_route_inserts_at_position_zero` — pins the
  `routes[0]` insertion semantics so a future refactor that changes the
  index fails this test.

## v2.11 follow-ups

- **Adoption** of `register_test_route` is opt-in — no existing tests need
  it today because the test environment runs without `frontend/dist`. Land
  the helper now so the next time a test trips on the catch-all (when
  `frontend/dist` IS built in CI), the fix is a one-liner.
- **C5 lint** currently scans only `tests/**`. If `app/**` ever grows
  request mocks (unlikely), expand the glob.
- The C5 lint flags `<x>.headers = {...}` in any test file unconditionally
  — if a future test has a legitimate reason to assign a plain dict
  (e.g. asserting on something that isn't `Request`), add to
  `_ALLOWLIST` with a `<rel>:<line>` key.
