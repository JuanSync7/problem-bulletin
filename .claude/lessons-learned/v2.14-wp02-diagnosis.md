# v2.14-WP02 (B3) — request-side contract pins for `*Body` schemas

## Scope

Direct carry-forward of the seven `*Body` / one-shot ad-hoc schemas
that v2.13-WP04 explicitly deferred. WP04 covered the 17
`*Create` / `*Update` pairs; this WP closes the remaining request-side
gap by pinning the action-payload schemas the frontend POSTs to
endpoints like `/tickets/{id}/transition`, `/assign`, `/comments`,
`/links`, `/watchers`, `/attachments` and `/auth/magic-link`.

Same invariant, same walk, same polarity as WP04 — closed schemas,
`schema.model_fields − consumer_referenced_names ⊆ excluded_fields`.

## Polarity per pair

All schemas are closed (no `extra="allow"`). Standard direction
applies: `schema ⊆ consumer references`.

`TicketAssignBody.expected_version` was the only candidate OCC token
and might have suggested an `excluded={"expected_version"}` row by
analogy with WP04's PATCH-schema `version` token. It does NOT need
exclusion here because the assign route uses kwarg fan-out (not the
`exclude={"version"}` + `setattr` loop pattern) and forwards the
field explicitly. The kwarg-name walk picks it up.

## Drift findings

**Zero drift detected.** All 7 pairs pass green on first run.

Why this is unsurprising: every one of these endpoints uses the
kwarg fan-out pattern (or, for the single auth/magic-link case, a
direct attribute access). The route handler enumerates each schema
field by name when calling the underlying service, so a missing
field would show up immediately at code-review time as "schema has
N+1 fields but the route only references N kwargs". The lint
formalises what was already enforced by code-review convention.

This matches the WP04 result: the codebase already has strong
request-side contract discipline because of the kwarg-fan-out idiom
on POST routes. The lint is a regression net, not active cleanup.
