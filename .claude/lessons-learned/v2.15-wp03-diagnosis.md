# v2.15-WP03 (C1) — category-c silent-on-non-2xx sweep — diagnosis

## Context

v2.14-WP04's diagnosis catalogued **28** category-c sites — page-level
`fetch()` calls where the only branch is `if (res.ok) { ... }` with no
`else`, so a non-2xx response is silently swallowed. v2.15-WP02 pinned
these into a structural bare-`catch{}` allow-list (36 entries, 24 in
ProblemDetail) so a future PR can't reintroduce the class. WP03 is the
SWEEP: convert the silent path to a user-visible error, delete the
corresponding allow-list entries, regression-test a representative
slice.

Baseline at WP03 start: backend **1429 P / 0 F / 5 skipped / 14 xfailed**,
frontend **252 P / 0 F**.

UX choice (per prompt's autonomous-decision authorization): the page's
**existing error surface** is reused where it makes sense; otherwise an
inline `setError`-style state is added with minimal display.

---

## Per-site action table

### ProblemDetail.tsx — 24 sites — UX = new `actionError` state + inline banner

ProblemDetail.tsx already had a page-level `error` state (line 491),
but it is rendered in a **full-page replace** layout: when `error`
is non-null, the entire problem view is replaced by a `<div
className="problem-detail__error" role="alert">` with a Retry button.
Routing action / hydration errors through that state would hide the
problem entirely on a transient comment-load 5xx — terrible UX.

Decision: **add a second state** `actionError: string | null` plus a
dismissible inline `<div className="problem-detail__action-error"
role="alert" data-testid="problem-detail-action-error">` banner
rendered above the main content. The page's pre-existing `error`
state is left untouched (still drives the full-page replace for
`fetchProblem` failures, which is correct UX for "problem can't load
at all"). A file-local `throwParsed(res, fallback)` helper threads
the unified envelope's `message` to `setActionError`.

The sub-component `CommentItem` (reply/edit catches) does not have
direct access to the parent's `setActionError`. Mechanical fix:
extend its props with `onActionError: (msg: string) => void` and
pipe `setActionError` through both call sites (top-level
`comments.map` and recursive `visibleReplies.map`).

### Submit.tsx — 3 sites — UX = unchanged (documented exception)

The v2.14-WP04 diagnosis (lines 47–49) explicitly marks all three
Submit.tsx category-c sites as **intentional silent-degrade**:

- Line 87 / `fetchCategories`: "intentional empty-state on failure"
  (user sees empty dropdown — UX is "category picker degraded").
- Line 101 / `fetchDomains`: same pattern, same intentional empty
  state.
- Line 183 / per-attachment upload loop inside `handleSubmit`:
  "failedFiles aggregated, toast shown at end" — already has a
  user-visible aggregated error surface via the existing
  `toast.show("Problem created but failed to upload: ...")`.

Per the WP03 prompt's "category-c-by-design exception" clause —
"e.g., 404 on optional fetch means 'skip'" — these three sites
qualify. **Allow-list entries retained**. The migration is not a
no-op: it preserves the existing aggregated toast surface (the
loop's bare `catch{}` rolls failures into `failedFiles`, which
becomes a single user-visible toast). Forcing a per-file
`setError(envelope.message)` would actually be a UX regression
because it would replace the aggregate.

**Per-file UX choice: Submit = unchanged. Three sites carried
forward to v2.16 as "review intentional empty-state UX" if product
ever wants to surface a category-load failure differently.**

### Search.tsx — 1 site — UX = unchanged (documented exception)

Line 352 / categories-dropdown bootstrap: same pattern as
Submit's category bootstrap. The dropdown is one of many search
filter inputs and "filter is empty when admin/categories errors"
is the existing intentional UX. The page's own `error` state
(`useSearchV2().error`) is for **search-query failures**, not for
sidecar filter-dropdown bootstrap — routing the categories
bootstrap through it would falsely tell the user their search
failed.

Per the WP03 prompt's "category-c-by-design exception" clause this
site qualifies. **Allow-list entry retained.**

**Per-file UX choice: Search = unchanged. Carried forward.**

## Regression tests added

`frontend/src/pages/__tests__/ProblemDetailActionError.test.tsx`
(4 tests):

1. **Solutions hydration 4xx surfaces envelope `message`** — mocks
   `GET /api/problems/p-1/solutions` returning 403 with
   `{error:{code:"forbidden",message:"Solutions are restricted for
   this project.",...}}`. Asserts the inline banner
   (`data-testid="problem-detail-action-error"`) renders the
   envelope's message. Pre-WP03 this would have been silently
   swallowed.
2. **Comments hydration 5xx surfaces envelope `message`** — same
   pattern with 503 and a `service_unavailable` envelope, proving
   the path is route-agnostic.
3. **Attachments hydration 4xx surfaces envelope `message`** — same
   pattern with 429 and a `rate_limited` envelope. Three hydration
   sites covered by per-test mocking — pinpoints any regression to a
   specific route.
4. **Banner dismisses on click** — proves the banner is a
   state-driven UI element, not a fire-and-forget toast. The user
   can clear it; the next action will re-set it.

Pre-WP03, all four tests would FAIL (the banner element does not
exist; non-2xx is silently dropped).

The four representative tests cover three call-site shapes:
hydration GET, action POST (implicitly — same code path), and the
dismissible-state UX contract. The remaining 21 migrated sites in
ProblemDetail.tsx all follow the same convergent shape
(`throwParsed` + `setActionError` in the catch); the lint pins the
absence of bare `catch{}`, and the test pins the wiring of
`setActionError` to the banner. The 21 sites are mechanically
identical to the 3 hydration tests and would only fail in the
extremely-narrow case where someone added a fourth surface (e.g.
toast) that didn't route through `actionError` — not worth a
per-site test.

Submit.tsx and Search.tsx received no regression tests in this WP
because their sites are documented exceptions (no code change).
Adding tests for them would just pin the existing intentional
silent-degrade behavior, which is the wrong contract: those tests
would fail if/when v2.16 product decides to surface a category-load
failure.

## Convergent pattern

Every migrated site shares the same five-line shape:

```ts
try {
  const res = await fetch(...);
  if (!res.ok) {
    await throwParsed(res, "fallback message");
  }
  // (original ok-branch logic, no longer guarded by `if (res.ok)`)
} catch (err) {
  setActionError((err as Error).message || "fallback message");
} finally {
  // (existing finally, if any)
}
```

For sub-component `CommentItem`, the catch uses the
`onActionError` prop instead of `setActionError` directly.

This shape is mechanically identical to the file-local
`throwParsed` pattern from v2.14-WP04's admin/*.tsx migration, and
to the inline `parseApiError` pattern from
`Feed.tsx`/`Leaderboard.tsx`. The convergence is the v2.13-WP03
lesson playing out across three WPs in a row: **a permissive
parser at the bottom + a thin convergent wrapper at every page is
the cheapest sweep shape**.

## Lessons (for v2.15 retrospective)

1. **Full-page-replace `error` is not the right channel for action
   errors.** ProblemDetail had a page-level `error` state, but
   reusing it would have replaced the entire problem view on a
   transient action failure. Adding a second `actionError` state
   with an inline banner is the minimal mechanical fix — one
   `useState`, one `<div>`, one CSS rule, no state restructure.
   Future page-level sweeps should check whether the page's
   existing error state is "load failure" vs "action failure"
   shaped before reusing it.

2. **Sub-components without state access need a callback prop, not
   a context.** `CommentItem` (which makes its own fetches) needed
   `onActionError`. A `useContext`-based `ActionErrorProvider`
   would have been overengineered; one prop is the smaller blast
   radius.

3. **The lint's stale-entry detection forced the allow-list
   cleanup.** As soon as a catch was promoted from `catch {}` to
   `catch (err) { ... }`, the line number shifted (or the entry
   was no longer bare), and the lint failed loud at both the new
   line (as a new offender — which we then quickly added if
   genuinely retained) and the old line (as a stale entry). Net:
   the only path to a green lint was to actually update the
   allow-list with every migration. This is exactly the
   forcing-function discipline v2.15-WP02's diagnosis claimed it
   would deliver.

4. **"Category-c-by-design" is a real category.** Three Submit
   sites + one Search site are intentional silent-degrades —
   "dropdown empty when admin endpoint errors" is the existing UX
   and shouldn't be replaced with an aggressive error banner.
   The prompt explicitly authorizes such exceptions; the diagnosis
   names each one so v2.16 can revisit if product wants stronger
   UX. The lint allow-list retains those entries with explicit
   justifications.

5. **Inline `alert()` was the most-replaced UX surface, not the
   bare catch.** The inline status-transition handler (line 1315
   pre-WP03 / 1390 post) had a hand-rolled `alert(data?.detail
   ...)` for the non-ok case — a UX inferior to the unified banner
   AND reading the **legacy** `detail` shape instead of the
   unified envelope `message`. WP03's banner subsumes it.
