## Production fixes (G2)

### `app/services/search.py` — `p.status` → `p.legacy_status`

The raw-SQL search query referenced `p.status = :status` when filtering
by problem status. The `problems` table column was renamed in alembic
migration `a1_agent_kanban_rename_problems_to_tickets`:

```python
op.alter_column("tickets", "status", new_column_name="legacy_status")
```

The ORM mapping (`Problem.status` → `legacy_status`) absorbed the rename
for ORM-based queries (e.g. `app/services/leaderboard.py:get_top_solvers`
filters on `Solution.status == "accepted"` and never breaks). But the
*raw* SQL string in `app/services/search.py:search_problems` was not
updated, so any call from `GET /search?status=open` would hit:

```
psycopg.errors.UndefinedColumn: column p.status does not exist
```

Workflow followed:

1. Wrote `test_search_filter_status` against the live DB (asserts open
   problem is matched, resolved one is not).
2. Ran the test red — `ProgrammingError: column p.status does not exist`.
3. Patched the raw SQL to use `p.legacy_status`; added a comment pointing
   at the migration that caused the asymmetry.
4. Re-ran — green.

Same shape as WP04a's discovery in seed helpers: ORM column-mapping makes
the rename invisible to model-bound queries, but raw SQL outside the
migration files needs to be swept manually. Documented again in
v2.11 follow-ups.

## v2.11 follow-ups discovered

1. **Audit raw-SQL references to renamed columns.** WP04b found one
   (`search.py`); WP04a's seed helper hit the same trap from the test
   side. A repo-wide grep `(p|problems)\.status\b` outside migration
   files would surface any remaining offenders. Better still: enforce
   the ORM column-mapping by deleting the `legacy_` prefix at the schema
   level, or rename the DB column back and migrate the ORM mapping.

2. **`get_tags` silently swallows invalid `sort` values.** The original
   v1 test asserted 422 on `sort=invalid`. Production silently falls
   back to name-order. If the public contract is meant to be strict,
   `get_tags` should validate `sort ∈ {"name", "usage"}` and raise
   422 — and the route should propagate the validation error. If the
   contract is "permissive", the docs should say so.

3. **`AION_BULLETIN_TEST_DOCS.md` cites in test docstrings still point at
   a phantom contract.** (Repeats WP04a's follow-up #2.) The original
   test_search.py docstring cited "lines 1603–1730" and test_admin.py
   referenced specific sections. Either reconcile that document or
   delete the citations from the surviving tests.

4. **`update_user_role` doesn't validate the new role string.** The
   service writes whatever value the caller passes (no `role IN ('user',
   'admin', …)` check at the service layer). Route-layer guard
   (`Annotated[str, Field(pattern=…)]`) is the only defence. The test
   suite pins the loose contract — tighten if the contract is meant
   to be strict.

5. **`update_config` uses `"admin"` as the literal `user_id` in the audit
   trail.** `log_event("config.updated", "app_config", key, "admin", …)`
   — a string sentinel rather than the real actor's id. Compared with
   `update_user_role` which passes `str(user_id)` (the *target* user id,
   not the admin's), the audit-actor story is inconsistent. The test
   only asserts that *something* about `"config.updated"` is logged.
   Worth a sweep.

## Surprises

- **One real production bug among 77.** WP04a was 0 / 91 bucket (b);
  WP04b is 1 / 77. The bug was reachable from a live HTTP route (it's
  the public search endpoint's status filter), so this is a genuine
  user-facing fix, not just a test-only contract drift.
- **The `caplog` fixture works against the `aion.events` logger** without
  needing a custom propagate hook — the `log_event` helper writes through
  `logging.getLogger("aion.events")`, which propagates to root. The tests
  use `caplog.at_level("INFO", logger="aion.events")` plus a `getattr` walk
  over `record.event_type` to be tolerant of the structured-extra encoding.
- **`get_tags` returns `list[dict]`, not `list[Tag]`.** The original v1
  test passed `MagicMock(name="...", usage_count=10)` and treated rows as
  attribute objects. Production aggregates via outer join and dict-packs
  the result — there is no `Tag.usage_count` column.
- **`de_anonymize` returns `{"author_id": UUID}`, not the bare UUID.** The
  original v1 test asserted `result == problem.author_id`. The route layer
  also expects the dict — the v1 tests would have caught no real bug here
  either way, but the test name was misleading.
- **`create_category`'s "empty table → sort_order=0" claim only holds on
  a clean DB.** Production reads `coalesce(MAX(sort_order), -1) + 1`, so
  the *exact* number depends on what's already in the dev DB. WP04b's
  test snapshots `MAX(sort_order)` before the create and asserts the new
  row's value is `max+1` — robust against pre-existing data.
- **The full-suite delta from "rewrite-only" was net negative for a
  beat.** We initially deleted the 20+4 non-deferred tests during the
  rewrite (they used `mock_db`). The full-suite count dropped from
  the expected 1103 to 1079 (= 1026 + 77 − 24). Restored by re-porting
  those non-deferred tests to live DB inside the same files (added
  `TestCreateCategoryExtras`, `TestUpdateConfig`, etc.). Lesson for
  WP04c: when rewriting a file, keep an inventory of which tests in it
  were *already passing* on mock_db and re-port them too, otherwise
  coverage silently regresses.
