# v2.16-WP02 (Z1) — SyntaxWarning + docstring escape audit

## Symptom

Baseline v2.15-WP01 emitted at import time:

```
SyntaxWarning: invalid escape sequence '\)'
  File "app/services/search_multi.py", line 119
    """Escape LIKE metacharacters (%, _, \) in user input ...
```

Under `python -W error::SyntaxWarning`, this is promoted to `SyntaxError` and
fails import. On Python 3.13+ this becomes the default. CI runs that capture
stderr also get noisy.

## Files fixed

| File | Line | Mechanism |
|---|---|---|
| `app/services/search_multi.py` | 119 | r-string prefix on docstring |

That was the only offender. Audit was performed by compiling every `.py`
under `app/` with `warnings.simplefilter('error', SyntaxWarning)` plus a
`pkgutil.walk_packages` + `importlib.import_module` sweep — both surfaced
zero remaining issues post-fix.

## Fix mechanism

The docstring `"""Escape LIKE metacharacters (%, _, \) in user input..."""`
contained a literal `\)` — Python does not recognise `\)` as a valid escape
and warns. We converted the docstring to a raw string by prefixing `r`:

```python
def _escape_like(value: str) -> str:
    r"""Escape LIKE metacharacters (%, _, \) in user input before building LIKE patterns.
    ...
    """
```

This is escape-only — zero behavioural change. `r"""..."""` is a docstring
just like `"""..."""`; it shows up in `__doc__` and Sphinx exactly the same.
The body of the function is untouched.

Alternative considered: escape to `\\)`. Rejected because the docstring is
documenting LIKE metacharacters; a reader expects to see `\)` (the actual
character pair), not `\\)`. Raw-string keeps the rendered docstring honest.

## Regression test

`tests/test_no_syntax_warnings_wp02_v216.py` — 3 cases:

1. `test_app_import_under_werror` — subprocess `python -W error::SyntaxWarning -c "import app"` exits 0.
2. `test_all_app_submodules_import_under_werror` — walks every submodule
   under `app.` via `pkgutil.walk_packages`, imports each, asserts no
   SyntaxWarning escapes (filters genuine ImportError from missing optional
   deps but FAILS loud on any escape issue). This is the test that catches
   lazily-imported leaf modules — `import app` alone does NOT reach
   `app.services.search_multi`.
3. `test_helper_catches_synthetic_bad_module` — self-test: writes `\)` into
   a tmp module, runs the same recipe, asserts the helper catches it. Guards
   against the regression test silently passing if the recipe regresses.

RED→GREEN was confirmed: before the r-string fix, case 2 reported three
failure chains:

```
app.main: invalid escape sequence '\)' (search_multi.py, line 119)
app.routes.search: invalid escape sequence '\)' (search_multi.py, line 119)
app.services.search_multi: invalid escape sequence '\)' (search_multi.py, line 119)
```

After the fix, all 3 cases pass.
