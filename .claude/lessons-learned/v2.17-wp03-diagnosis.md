## Implementation notes

### Why regex over `splitlines()`, not AST

The `ast` module **strips comments during parse**. `ast.parse` does
not preserve any comment-pragma — `# type: ignore` and `# noqa` are
both invisible from an AST walk. A `tokenize`-based approach could see
them but adds complexity for no win: the patterns are line-level
markers, and a regex over `Path.read_text().splitlines()` is
straightforward and sufficient.

Patterns used:

- `r"#\s*type:\s*ignore\b"` — matches `# type: ignore`,
  `#type: ignore`, `#  type:  ignore`, and any `[code]` suffix
  (`\b` matches at the `[`).
- `r"#\s*noqa\b"` — matches `# noqa`, `#noqa`, `# noqa: F401`,
  `# noqa: BLE001 - reason`.

### Edge cases

1. **String-literal false positives.** A line like
   `s = "# type: ignore in docstring"` would falsely flag. **No such
   site exists in `app/` today** — verified by inspecting all 34
   matches. If a future refactor introduces one, the simplest fix is
   to add the line to `_OFFENDER_ALLOWLIST` with a one-line
   `BY-DESIGN: string-literal false positive` justification; for a
   broader sweep, switch the implementation to `tokenize` and filter
   to `tokenize.COMMENT` tokens.
2. **Multiple pragmas on one line.** A single line can match both
   `# type: ignore` and `# noqa` — `_scan_lines` returns two tuples
   in that case. The allow-list keys on `path:line` only, so a single
   dict entry suffices for the whole line. No such site exists today.
3. **Allow-list-key stability across whitespace edits.** The key is
   `path:line` only, so renaming a variable on an offender line keeps
   the entry valid. Inserting a new line above does shift line
   numbers — the stale-detection branch fails loud, which is the
   intended forcing function for the maintainer to re-check the
   shifted entry.
4. **The lint file scans itself but is filtered.** Not relevant here
   because the lint lives under `tests/` and only `app/**/*.py` is
   scanned. No self-exclusion needed.
5. **Comment continuations / multi-line strings.** Python comments
   are single-line by definition, so a `#`-marked pragma cannot span
   lines. `splitlines()` is safe.

### Scope exclusions

- `tests/**` — out of scope (test-only suppression patterns differ).
- `alembic/**` — lives at repo root, not under `app/`. The `rglob`
  over `app/` naturally excludes it. The lint also carries a
  defensive `_EXCLUDED_PREFIXES = ("app/migrations/", "app/alembic/")`
  in case a future refactor moves alembic under `app/`.
- `scripts/**` — out of scope (one-off utilities).
