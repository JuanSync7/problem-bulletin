# v2.13-WP03 (A2) ŌĆö frontend parseApiError sweep ŌĆö diagnosis

## Goal

Migrate every ad-hoc error-parsing branch under `frontend/src/api/` to the
unified `parseApiError` adapter introduced by v2.12-WP09. The adapter is
permissive: it accepts both the new `{error:{code,message,correlation_id,details}}`
envelope and the legacy `{detail: ...}` shape. No behaviour change for the
already-migrated path; latent legacy-shape responses now also produce a
useful message instead of `undefined`/`HTTP N`.

## Migration pattern

All seven sites converged on the same shape:

```ts
if (!res.ok) {
  const body = await res.json().catch(() => null);
  const parsed = parseApiError(res, body);
  const env: ErrorEnvelope = {
    code: parsed.code,
    message: parsed.message,
    details: (parsed.details ?? undefined) as Record<string, unknown> | undefined,
    correlation_id: parsed.correlation_id ?? undefined,
  };
  throw new ApiError(res.status, env);
}
```

`ApiError`/`ErrorEnvelope` are reused from `tickets.ts` so UI consumers see a
single class. The throw shape is unchanged ŌĆö every existing `instanceof
ApiError` check and `.envelope?.code` access keeps working.

`audit.ts` preserves its `res.status === 404 ŌåÆ []` early-return; only the
error-parsing branch was touched.

`users.ts` is untouched: it already routes through `parseApiError` and
populates its custom `UpdateHandleError` subclass (with `next_allowed_at`
extracted from `parsed.details` and falling back to the legacy top-level
key for back-compat).

## Page-level inline fetches (DEFERRED)

The grep surfaced ~60 inline `fetch()` call sites in `frontend/src/pages/`,
mostly hitting the **pre-ticketing legacy surface** (`/api/problems/*`,
`/api/admin/*`, `/api/comments/*`, `/api/solutions/*`). Per WP rule 4 ("If
the page does its own fetch + parse, that's in scope IF it's parsing
errors") and the WP scope ("focus on the API layer where errors first
cross the boundary"), these were **NOT** migrated in WP03:

- They are legacy (pre-WP09) routes, mostly admin/problem-bulletin domain,
  and the unified envelope rollout was scoped to the ticketing surface.
- Most ignore the error body entirely (e.g. `if (res.ok) setData(...)`),
  not "parse the error and throw" ŌĆö so there is nothing to route through
  the adapter without a behaviour change.
- Two sites read `.detail` from a legacy body (`Settings.tsx:380`,
  `ProblemDetail.tsx:1305`); `Settings.tsx` reads from
  `UpdateHandleError.detail` which is already populated by the adapter,
  and `ProblemDetail.tsx:1305` is a one-off inline status-transition
  alert against `/api/problems/{id}/status` (legacy surface).

Recommendation: punt to v2.14 ŌĆö track as "page-level inline-fetch
legacy-surface sweep" once the legacy `/api/...` routes are themselves
migrated to the unified envelope.
