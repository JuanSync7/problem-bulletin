# v2.10-WP06 — Diagnosis of the "4 deferred alembic-roundtrip tests" cluster

## TL;DR

**Classification: 2 × bucket (a) + 2 × bucket (c).** One file-level fixture
drift (the `alembic` CLI is not on PATH inside the venv) plus a stale
hard-coded head revision masked a real production migration bug: the
auto-generated downgrade in `7f57993c9b09_add_domains_table_and_domain_id_to_.py`
called `op.drop_constraint(None, 'problems', type_='foreignkey')`, which is
unrunnable — alembic raises `CompileError: Can't emit DROP CONSTRAINT for
constraint ... it has no name`. The corresponding `op.create_foreign_key(None, ...)`
in the upgrade is the upstream defect.

Once the fixture was repaired and the head expectation made
revision-agnostic, the chain-walks-to-base test caught the production bug
that the original 4-step downgrade had been silently sidestepping.

## Production fix (bucket c)

### `alembic/versions/7f57993c9b09_add_domains_table_and_domain_id_to_.py`

**Defect.** The auto-generated stub left the FK name as `None` on both
upgrade and downgrade:

```python
# upgrade
op.create_foreign_key(None, 'problems', 'domains', ['domain_id'], ['id'])
# downgrade
op.drop_constraint(None, 'problems', type_='foreignkey')
```

`create_foreign_key(None, ...)` defers naming to postgres, which assigns
`problems_domain_id_fkey` from its default naming convention. The
downgrade then tries to emit `DROP CONSTRAINT` on an unnamed
SQLAlchemy `ForeignKeyConstraint`, which compiles to nothing and raises
`CompileError`.

**Fix.** Give the FK an explicit, stable alembic name
(`fk_problems_domain_id_domains`) on upgrade and drop by name on
downgrade. To handle databases already upgraded before this rename, use
raw SQL `ALTER TABLE problems DROP CONSTRAINT IF EXISTS ...` for *both*
the new name and the legacy auto-assigned `problems_domain_id_fkey`, on
both sides of the migration. This is idempotent and safe to re-run.

### Red regression first (G2)

`tests/migrations/test_domains_fkey_downgrade.py` was written before the
production edit and parses the migration AST to assert no
`drop_constraint(None, ...)` or `create_foreign_key(None, ...)` calls
remain. It is a static check (no DB required), so it runs in every CI
environment — not just where postgres is reachable. The dynamic
live-DB cycle is covered by `test_each_agent_kanban_revision_is_reversible`
which now walks to `base` rather than 8 steps down.

## v2.11 follow-ups discovered

1. **Sweep alembic versions for `op.create_foreign_key(None, ...)` and
   other unnamed-constraint auto-stubs.** The auto-generator emits
   `None` for the constraint name by default; only the downgrade reveals
   it. Add a project-wide lint (extend the static AST test in
   `tests/migrations/test_domains_fkey_downgrade.py` into a sweep over
   `alembic/versions/*.py`) that flags any `None`-named constraint
   creation across the whole tree.
2. **Set a SQLAlchemy `naming_convention` on the project `MetaData`.**
   That would have made postgres assign `fk_<table>_<col>_<reftable>`
   automatically and rendered the `None` downgrade harmless.
   `MetaData(naming_convention={...})` plus `target_metadata` wiring in
   `alembic/env.py` is a one-time project-wide fix and prevents future
   stubs from sliding in with this defect.
3. **Drop or refactor `AGENT_KANBAN_REVS` constant.** It's a stale
   hand-maintained list (only a1..a8) and the test no longer uses it
   for its step count. Either delete or repurpose as an assertion that
   each of a1..a8 *individually* appears in `alembic history`.
4. **Hard-coded head string in two tests was a maintenance footgun.**
   Generalise the `_current_head_revision()` helper out of this file
   into a shared `tests/migrations/_helpers.py` for the next time
   someone writes a head-pinned smoke test.

## Surprises

- The 4-ID cluster was sold as a "fixture drift" cluster like WP03, but
  ID #2 hid a real (c) production bug behind a too-short downgrade
  loop. The bug had been silently latent since the domains migration
  was added (Apr 2026) — no environment had ever actually downgraded
  past `7f57993c9b09`.
- The original test's downgrade-by-N pattern was a maintenance smell:
  every new revision quietly weakened the test, because the same N
  steps from a moving head reached a progressively later floor.
  Switching to `downgrade base` decouples the test from the chain
  length and is the correct shape for a roundtrip test.
- `924` not the projected `922` because the WP02 G2 regression test
  added two static AST assertions, not just one as roughly counted
  from the brief's `918 + 4` math.
