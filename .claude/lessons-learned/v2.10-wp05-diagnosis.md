# v2.10-WP05 — Diagnosis of the "11 deferred Settings tests" cluster

## TL;DR

**Classification: 2 × bucket (a) + 9 × bucket (b).** Two "default value"
tests were pure ambient-env leakage: `tests/conftest.py` calls
`os.environ.setdefault("APP_NAME", "Aion Bulletin Test")` and
`STORAGE_PATH=/tmp/aion-test-storage` at import time, and pydantic-settings
v2 *always* layers OS env on top of explicit constructor kwargs unless the
env keys are removed. The remaining nine "missing required field" tests
exposed a real bucket-(b) production drift: during the v1→v2 migration the
`Settings` model in `app/config.py` had grown `= ""` / `= SecretStr("")` /
`= "http://localhost:8000"` defaults on AAD, JWT, SMTP and BASE_URL fields
that the contract (REQ-104, REQ-108, REQ-504) says must be supplied
explicitly. With defaults present, `Settings()` with no inputs constructed
silently — exactly the boot-time misconfiguration the contract was meant
to catch.

The 11 deferred IDs were a single TDD-shaped red regression: the
contract tests already existed (deferred since WP01); promoting them
to green required restoring the "no default" production posture for
the six contract-required fields plus an env-isolation fixture for the
two default-value tests.

## Production fix (bucket b)

### `app/config.py`

**Defect.** The `Settings` model carried v1-vintage placeholder defaults
on six contract-required fields:

```python
AZURE_TENANT_ID: str = ""                  # REQ-504
AZURE_CLIENT_ID: str = ""                  # REQ-504
AZURE_CLIENT_SECRET: SecretStr = SecretStr("")  # REQ-504
JWT_SECRET: SecretStr = SecretStr("dev-secret-change-me")  # REQ-108
SMTP_HOST: str = ""                        # REQ-104
SMTP_FROM: str = ""                        # REQ-104
BASE_URL: AnyHttpUrl = "http://localhost:8000"  # REQ-104
```

Effect: a misconfigured deployment (no AAD env vars, no JWT_SECRET,
no SMTP, no BASE_URL) booted successfully and only failed at first
auth/email/link-generation attempt. The REQ-104/108/504 contract is
that these must be supplied at boot or the process exits.

**Fix.** Removed the placeholder defaults from all six fields plus
the `JWT_SECRET` `dev-secret-change-me` footgun. The fields are now
strictly required: omitting any one raises `ValidationError` at
`Settings()` construction.

### Red regression first (G2)

The 11 deferred tests in `tests/test_config.py` (added in v1, deferred in
WP01) *are* the red regression. They were promoted in this WP. Before
the production edit, the 9 "missing-required" cases reported
`DID NOT RAISE`; after the edit, all 9 raise `ValidationError`. No new
test file was needed — WP05 is the "promote the existing contract" shape
of TDD, not the "write a new test" shape.

## Per-file edit summary

### `app/config.py`
- Removed `= ""` defaults from `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`,
  `SMTP_HOST`, `SMTP_FROM`.
- Removed `= SecretStr("")` default from `AZURE_CLIENT_SECRET`.
- Removed `= SecretStr("dev-secret-change-me")` default from `JWT_SECRET`
  (dev-secret was a v1 footgun masquerading as a default).
- Removed `= "http://localhost:8000"` default from `BASE_URL`.
- Added inline comments tagging each field as REQ-104 / REQ-108 / REQ-504
  required.
- All other defaults (APP_NAME, ENVIRONMENT, STORAGE_PATH, SMTP_PORT,
  DEV_AUTH_BYPASS, TEAMS_WEBHOOK_URL, OTel block, scanner/retention
  blocks) are unchanged — those are genuine optional knobs.

### `tests/test_config.py`
- New `_SETTINGS_ENV_KEYS` tuple enumerating every env var the Settings
  model reads.
- New `_isolated_env` `monkeypatch`-based fixture that calls
  `monkeypatch.delenv(key, raising=False)` for each of those keys. This
  strips the ambient `conftest.py` env so explicit constructor kwargs
  become the only source of truth.
- `_make_settings()` updated to pass `_env_file=None` so a developer's
  local `.env` cannot leak in.
- The two default-value tests (`test_default_app_name`,
  `test_default_storage_path`) and all 9 missing-required tests now
  depend on `_isolated_env`.
- Missing-required tests also pass `_env_file=None` directly to
  `Settings(...)` (these tests bypass `_make_settings` because they
  intentionally build incomplete kwargs).

### `tests/_v1_deferred.py`
- Removed the 11 IDs in scope. New count: 254 (was 265).

## v2.11 follow-ups discovered

1. **Add a boot-time fail-fast for `ENVIRONMENT=production` plus
   `DEV_AUTH_BYPASS=True`.** The settings layer correctly stays neutral
   on this combination (per `test_production_with_dev_auth_bypass_no_settings_error`),
   but the caller-side check noted in that test ("Safety enforcement is
   the caller's responsibility") is not actually wired anywhere I can
   find. Add a startup hook in `app/main.py` that refuses to boot the
   process when both flags are on.
2. **`DATABASE_URL` async-driver enforcement.** Test file calls this
   out as a GAP (line 248). Add a `@field_validator` that asserts the
   URL starts with `postgresql+asyncpg://` (or `sqlite+aiosqlite://`
   for tests) and rejects sync drivers explicitly.
3. **`.env` file resolution order test.** Test file flags this as a
   GAP (line 250). Worth a small fixture-driven test that writes a
   `.env` to a tempdir and asserts precedence (OS env > kwargs > .env >
   model default).
4. **Sweep `conftest.py` for ambient-env footguns.** The two `(a)`
   defaults in WP05 were the visible tip; any test that asserts a
   model default for a `Settings` field could be masked by an
   `os.environ.setdefault(...)` line up top. Lint candidate: every
   key in `conftest.py`'s setdefault block must be either (i) overridden
   by every test that reads that field's default, or (ii) documented
   as load-bearing.
5. **Remove the dev-secret footgun from any leftover docs.** `JWT_SECRET
   = "dev-secret-change-me"` was deleted from `app/config.py`; verify
   no `.env.example` or README still shows that string as a safe
   default. (Not checked here.)

## Surprises

- Brief labelled WP05 as a "Pydantic-v2 settings rewrite" anticipating
  v1 `@validator` / `BaseSettings` / `Config` drift. Reality: `app/config.py`
  was *already* on `pydantic-settings` v2 with `@field_validator` and
  `SettingsConfigDict` — the v1→v2 migration had happened. The actual
  drift was subtler: defaults that had been "OK for dev" silently
  weakened the contract.
- The bucket-(a) and bucket-(b) layers were entangled: even after
  the production fix, 9 of the 9 "missing-required" tests *still*
  fail without the env-isolation fixture, because `conftest.py`
  provides every required env var as a fallback. You need both
  fixes; either alone is insufficient. The diagnosis bucket count
  reflects that compound shape.
- pydantic-settings v2 has no documented way to disable OS env
  reading for a single `Settings(...)` call. `_env_file=None` only
  disables the dotenv file. The only working pattern is to
  `monkeypatch.delenv` for the duration of the test — i.e., the
  test isolates *itself* from the harness, rather than the harness
  exposing an opt-out. That's worth a one-liner in the
  `pydantic-settings` cookbook section of the engineering guide.
