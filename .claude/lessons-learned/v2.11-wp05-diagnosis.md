# v2.11-WP05 — Boot/config hardening diagnosis

Three independent boot/config hardening items addressed in a single WP.

## A8 — Production fail-fast on `DEV_AUTH_BYPASS=True`

**Decision: implement in `app/main.create_app()`, not in `app/config.Settings`.**

The settings layer is intentionally neutral — there is already an explicit
regression test (`tests/test_config.py::TestEnvironmentLiteral::
test_production_with_dev_auth_bypass_no_settings_error`) pinning that a
production env + bypass=True does NOT raise at settings construction time.
The conftest sets `DEV_AUTH_BYPASS=false` and `ENVIRONMENT=development`
project-wide so most test fixtures are immune. Putting the guard in
`create_app()` keeps the settings constructor a pure data structure.

Implementation (`app/main.py`):

```python
if settings.ENVIRONMENT == "production" and settings.DEV_AUTH_BYPASS:
    raise RuntimeError(
        "DEV_AUTH_BYPASS must be False when ENVIRONMENT=production. ..."
    )
```

Edge cases considered:
- Module-level `app = create_app()` at the bottom of `app/main.py` means
  the guard fires at import time when env is bad. Test imports
  `create_app` BEFORE flipping env to avoid that import-time raise.
- Development + bypass=True (the common dev mode) — boots, no change.
- Staging + bypass=True — boots; only production is guarded.
- Production + bypass=False — boots (regression guard against
  over-tightening).
- `app = create_app()` at module level: in CI/dev the env is benign, so
  this is fine. In production, both module load AND any explicit
  `create_app()` call will raise, which is the desired behaviour.

## A9 — `DATABASE_URL` async-driver enforcement

Added `@field_validator("DATABASE_URL")` to `app/config.Settings`.
Accepts `postgresql+asyncpg://` and `sqlite+aiosqlite://`. Rejects:
- `postgresql://` (bare sync)
- `postgres://` (bare sync alias)
- `postgresql+psycopg2://` (explicit sync driver)
- `sqlite:///` (sync sqlite)

Edge cases considered:
- conftest.py uses `postgresql+asyncpg://test:test@localhost:5432/test_aion`
  — already async, no change required.
- `tests/test_config.py::REQUIRED_ENV` also uses the asyncpg URL.
- No call site in the codebase uses sqlite (live DB only), but
  `sqlite+aiosqlite://` is whitelisted in case a future WP adds an
  in-process integration test variant. If sync sqlite is ever needed,
  the validator's error message points the operator at the canonical
  async URL formats.
- Error message includes the offending scheme prefix so misconfigs are
  diagnosable from a single log line.

## A10 — `_EXCEPTION_STATUS_MAP` dead-entry removal

The central map in `app/main.py` declared `ForbiddenTransitionError → 409`
and `ForbiddenError → 403`, but the tickets module's
`EXCEPTION_HANDLERS` (registered AFTER the generic handler) overrides
both:
- `ForbiddenTransitionError → invalid_transition_handler → 422 envelope`
- `ForbiddenError → forbidden_handler → 403 envelope`

The per-route registration always wins. Removed both entries from the
central map. Added a regression test that enumerates the map and asserts
no class also has a module-local override.

Edge cases considered:
- `ForbiddenError`'s tickets-local status is the same (403) as the
  central map's, BUT the envelope format differs (plain `{"detail": ...}`
  vs. the tickets `{"error": {...}}`). Keeping both would be a latent
  bug: any non-ticket surface raising `ForbiddenError` would now get the
  tickets envelope. The override-wins rule is enforced consistently.
- Did NOT unify the envelope in this WP — flagged as a follow-up for a
  future "exception envelope normalisation" WP.
- The pinned regression test `test_forbidden_transition_live_override
  _still_returns_422` continues to assert that the production behaviour
  (422 invalid_transition envelope) is unchanged.
