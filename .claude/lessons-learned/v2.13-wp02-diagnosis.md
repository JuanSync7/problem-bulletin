# v2.13-WP02 (Bucket A1) — ck naming_convention resurrection: diagnosis + sweep inventory

## Background

v2.11-WP10 shipped 4 of 5 keys in `Base.metadata.naming_convention`
(`ix`, `uq`, `fk`, `pk`). The `ck` key was deferred because every
pre-WP10 `CheckConstraint(name=...)` in the repo passed the FULL
already-prefixed name (`name="ck_<table>_<short>"`); the idiomatic
template `"ck_%(table_name)s_%(constraint_name)s"` would double-wrap
those at DDL compile time
(`ck_<table>_ck_<table>_<short>`).

v2.12-WP08 attempted the resurrection mid-flight and was rate-limited
partway through. Its surviving artifacts:

- All `app/models/*.py` `CheckConstraint(name=...)` ports done to bare
  short names (e.g. `name="key_format"` not `name="ck_projects_key_format"`).
- 5 alembic migration files (a1, a2, a3, a10, a18) wrapped with
  `sqlalchemy.sql.elements.conv(...)` so the on-disk constraint name
  string short-circuits the convention substitution.
- `app/database.py` still has the `ck` key COMMENTED-OUT (deferred).
- The WP10 lint `tests/test_naming_convention_wp10.py` still asserts
  `ck` is OMITTED from the convention (line 39-48).

## Strategy chosen (WP02 v2.13)

**Wrap-with-`conv()` everywhere in alembic, keep bare names in models.**

Rationale:
- Model files already use bare names (v2.12-WP08 surviving port).
- Alembic migration history is immutable in production — porting
  literal name strings to "bare" form would change the names that
  alembic emits to postgres on a fresh upgrade run AND would mismatch
  the names that production postgres already has, requiring a live-DB
  `RENAME CONSTRAINT` migration to reconcile.
- The `conv()` wrapper short-circuits the convention substitution at
  DDL render time (verified by SQLAlchemy 2.x `sqla_compat`):
  `conv("ck_<t>_<s>")` produces `CONSTRAINT ck_<t>_<s> CHECK (...)`,
  not `ck_<t>_ck_<t>_<s>`.
- This means **NO live DB rename is required**: the literal full name
  emitted by alembic equals the postgres-side constraint name.
- The WP10 lint `test_alembic_constraint_name_lint_wp10.py` continues
  to pass — `conv(...)` is a `Call(func=Name)`, not a `Constant(None)`,
  so the `None`-name scanner is unaffected.

The competing strategy ("port migration name literals to bare short
names + write live-DB rename migration") was the prompt's preferred
approach, but I rejected it for two reasons:

1. **Migration history immutability.** Rewriting historical literal
   names in alembic versions is a higher-risk change than wrapping them
   with a render-time no-op — and the prompt's own G8 footnote
   ("likely zero pairs given G3+G4's plan") admits the rename is
   probably a no-op marker.
2. **Symmetry with v2.12-WP08 partial.** The 5 files already wrapped
   establish the project convention; finishing the sweep keeps the
   codebase internally consistent.

The G8 "rename migration" is therefore a comment-only marker (no
ALTER TABLE statements emitted) — documented below.

## Live-DB rename migration (G8)

Per the analysis above, NO actual `ALTER TABLE RENAME CONSTRAINT` is
needed: the model-emitted name (`ck_<t>_<s>` via convention) and the
on-disk name (also `ck_<t>_<s>`, preserved via `conv()` in alembic)
already match. The migration is a no-op marker
(`a20_ck_convention_alignment`) that documents the
"ck convention live as of this revision" boundary so a future grep
can find when the lever flipped.
