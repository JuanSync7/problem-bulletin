# v2.16-WP03 (Z2) — React Router v7 future-flag opt-in

## Summary
React Router v6.23 was emitting v7 future-flag warnings on every render that
mounted a Router. Two flags were warning:

- `v7_startTransition` — RR will wrap state updates in `React.startTransition` in v7.
- `v7_relativeSplatPath` — splat-route relative resolution changes in v7.

Both are forward-compat opt-ins. Adopting them in v6 changes nothing about app
behavior that we can observe from tests (no splat-based relative `useNavigate`
calls, no test asserting `startTransition`-specific timing).

## RR API in use
- Production: `<BrowserRouter>` (in `frontend/src/App.tsx`).
- Tests: `<MemoryRouter>` (16 test files).
- No `createBrowserRouter` / `RouterProvider` in the codebase.

`react-router-dom` pinned at `^6.23.1` in `frontend/package.json`. Version
unchanged in this WP.

## Flags enabled
Both flags supported by v6:
```ts
future={{ v7_startTransition: true, v7_relativeSplatPath: true }}
```
The other v7_* flags (`v7_fetcherPersist`, `v7_normalizeFormMethod`,
`v7_partialHydration`, `v7_skipActionErrorRevalidation`) apply only to data
routers (`createBrowserRouter` / loaders / actions) which we don't use, so they
never warned and don't need to be set.

## Before / after (vitest stderr)

### Before
Sample (warnings repeated ~per test file render):
```
⚠️ React Router Future Flag Warning: React Router will begin wrapping state updates in `React.startTransition` in v7. You can use the `v7_startTransition` future flag to opt-in early. ...
⚠️ React Router Future Flag Warning: Relative route resolution within Splat routes is changing in v7. You can use the `v7_relativeSplatPath` future flag to opt-in early. ...

Match count via `grep -ciE "v7_|future flag|React Router Future"`:
**non-zero (many tens of repeats)**.

0
```

## Regression test added
`frontend/src/__tests__/router_future_flags.test.tsx` — two cases:
1. With `future={...}` set, neither `console.warn` nor `console.error` is called
   with any string matching `/v7_|future flag|React Router Future/i`.
2. Sanity check: without the `future` prop, the warning IS emitted. This proves
   the positive assertion above isn't a no-op.

## Results
- Frontend test count: **256 → 258** (2 new regression cases). All green.
- Frontend warning count (`v7_|future flag|React Router Future`): **many → 0**.
- `npm run build` exits 0 — production build unaffected.

## Why "fix each test file" instead of a global mock
Considered (and rejected) a `vi.mock("react-router-dom", ...)` shim in
`src/test/setup.ts` that auto-injects the future prop. Rejected because:
- A global mock obscures the source of truth (a reader of a test wouldn't see
  the flag and wouldn't know it's set).
- Risk of subtle drift between mocked and real router behavior.
- The mechanical edit was small and 100% local — each MemoryRouter literally
  shows the flags it opts into.

## Forward-compat verification
No test asserts on `startTransition`-specific timing or on splat-relative
navigation paths. All 256 prior tests still pass unchanged, confirming the opt-in
is behaviorally inert for this codebase.
