## E1 — v1 `/api/search` sunset

### Choices

- **Sunset date:** `Sun, 22 Jul 2026 00:00:00 GMT` (RFC 1123). ~60 days
  from the deprecation landing (2026-05-22). Constant exported as
  `app.routes.search.V1_SEARCH_SUNSET_RFC1123` so tests + log
  consumers can grep for the same literal.
- **Header convention:** chose the `response.headers[k] = v`
  mutation pattern that already exists at
  `app/routes/tickets.py:76` (`_attach_corr`). Promoted to a
  module-level docstring inside the handler — RFC 8594 reference
  baked in.
- **Logger convention:** `logging.getLogger(__name__)` to match
  `app/routes/realtime_ws.py:39`, `app/routes/ws.py:14`,
  `app/routes/ws_tickets.py:22`. No structlog adoption in the routes
  layer; `app/logging.py::log_event` is reserved for audit-style
  domain events (user/role lifecycle), not surface deprecation. Plain
  `logger.warning(...)` is the right tool here.
- **Log format:** `"v1_search.hit caller=%s q_len=%d"`.
  - `v1_search.hit` is the grep tag — chosen because it parallels
    Datadog-style `<surface>.<event>` keys we already use in
    `connection_manager` logs.
  - `caller=auth:<scheme>` when `Authorization` is present
    (credential value NEVER logged — first whitespace/dot prefix
    only, truncated to 16 chars).
  - `caller=ip:<host>` when anonymous and a `request.client` is
    available.
  - `caller=unknown` fallback (defensive — never raises).
  - `q_len` rather than `q` itself: query strings can contain PII /
    sensitive search terms; the hit count is the metric we care about
    for the monitoring window, not the content.
- **No audit-log entry.** `app/logging.py::log_event` is structured
  around `(entity_type, entity_id, actor_id, action)` — a stateless
  hit-counter on a deprecated public surface does not map onto that
  shape, and adding a synthetic entity would dilute the audit table.
  Log-only is sufficient for the 60-day monitoring window.
- **Handler retained.** Per brief: instrumentation only. v1 still
  serves traffic until the monitoring window closes.

## E5 — `send_email_digest` empty-list contract

### What was undocumented

`app/services/delivery.py::send_email_digest` had a bare
`if not notifications: return` at line 142–143 with no docstring
explanation. Callers could not learn from `help()` whether passing
`[]` is a contract or an implementation detail subject to change.

### What changed

Extended the existing one-line docstring to a multi-line contract
note: empty list → returns `None` **before** DB lookup AND **before**
SMTP call. Cross-references the pinning test name so future maintainers
can trace contract → test → code.

### Test strategy

`tests/services/test_send_email_digest_empty_contract_wp13.py` —
**pure unit, no live DB**:

- `db` is a `MagicMock()` with `db.execute` / `db.flush` patched to
  `AsyncMock()`. We assert `.assert_not_called()` on both — a single
  await on `db.execute` would have been caught.
- `aiosmtplib.send` is patched at module level (`patch("aiosmtplib.send",
  new_callable=AsyncMock)`); we assert `.assert_not_called()`.
- Return value asserted to be `None`.
- Extra test passes an invalid UUID string to confirm the early
  return runs *before* `uuid.UUID(user_id)` parsing — guards against
  someone "fixing" the early-return by moving it after the user
  lookup.
- Docstring-content test asserts the contract is documented (so the
  docstring can't silently regress to a one-liner).

### Background-task wrapping

`send_email_digest` is **not** wrapped in a Celery / background task
chain in this codebase — `grep -rn send_email_digest app/` returns
only the definition site at `app/services/delivery.py`. No additional
docstring placement was needed.

## Lessons / pitfalls

1. **`caplog` + JSON logging config:** the project wires a JSON
   handler that replaces propagation paths; `caplog.records` came
   back empty even though the log message appeared in stdout. The
   robust pattern (now used in this WP and worth promoting to a
   helper) is to attach a fresh `logging.Handler` directly to the
   target logger inside the test and detach it in a `finally:`. Done
   inline here; a shared helper at `tests/helpers/log_capture.py`
   would be a useful follow-up.
2. **Bare `FastAPI()` lint (v2.11-WP09):** the first draft of the
   route test used `FastAPI()` + `include_router(...)` and tripped
   the `test_no_bare_fastapi_constructor_outside_allowlist` lint.
   Resolution: switch to
   `tests.helpers.app_factory.build_test_app(dependency_overrides=...)`.
   Worth surfacing in the v2.11 lessons file so subsequent subagents
   don't repeat this pattern.
3. **Authorization log hygiene:** the caller resolver truncates the
   token scheme/prefix to 16 chars and never logs the credential
   itself. The dedicated test
   `test_v1_search_logs_auth_scheme_when_authorization_header_present`
   pins both halves (scheme captured AND credential excluded) so
   future "improvements" to the log line can't silently leak tokens.
