| `test_tenant_mismatch_error_is_403` | PORT | ditto |
| `test_unmapped_appError_returns_500` | PORT | Pins fallback when an `AppError` subclass isn't in `_EXCEPTION_STATUS_MAP`. |

**Root cause of failure:** `app.main.create_app()` registers an SPA
catch-all `@app.get("/{full_path:path}")` LAST when `frontend/dist/` exists.
The v1 test helper `_make_exception_route` calls `create_app()` then
appends `/_raise_test` (and `/_unmapped`) — so those routes are registered
**after** the catch-all. FastAPI matches in registration order: the SPA
route wins, returns `index.html` with status 200, and the exception
handler is never reached.

**Port strategy:** insert the test route at the **front** of
`app.router.routes` so it precedes the catch-all. Equivalent to building
a minimal `FastAPI()` with only the `AppError` handler attached, but the
existing pattern asserts the real `create_app()` wiring, so we keep it
and just fix the ordering.

## G4 — Deferral mechanism

After this WP the manifest is empty. **Decision:** delete
`tests/_v1_deferred.py` AND remove the `pytest_collection_modifyitems`
hook from `tests/conftest.py`. Rationale:

- An empty manifest plus a 17-line hook is a strictly-worse "leave a
  trap for the next engineer" pattern than just removing both. Anyone
  reading `conftest.py` would have to grep for `V1_DEFERRED_NODE_IDS`
  to discover the hook is now inert.
- If a future WP needs per-ID deferral again, `xfail` / `skip`
  markers per-test are the idiomatic pytest way; we don't need a
  bespoke registry.

## Surprise — ForbiddenTransitionError has a domain override

The brief expected `ForbiddenTransitionError` → 409 per
`_EXCEPTION_STATUS_MAP`. Reality: `app.main` later calls
`app.add_exception_handler(ForbiddenTransitionError,
invalid_transition_handler)` via the ticket-domain
`EXCEPTION_HANDLERS` dict (`app/routes/tickets.py`). Per-route
handlers shadow the `@app.exception_handler(AppError)` umbrella, so
the production response is:

- status `422`
- body `{"error": {"code": "invalid_transition", "message": ...,
  "details": {}, "correlation_id": ...}}` — **no `detail` key**

The three parametrised v1 tests (`returns_correct_status_code`,
`response_body_has_detail_key`, `detail_contains_message`) plus the
single `test_forbidden_transition_error_is_409` were all asserting
the now-shadowed contract. Resolution: removed the `(ForbiddenTransitionError, 409)`
parametrise tuple from the three parametrised mappings and replaced
`test_forbidden_transition_error_is_409` with
`test_forbidden_transition_error_uses_ticket_envelope`, which pins
the real 422 + envelope contract. Three parametrised IDs are
collapsed into one focused assertion; that's still PORT (intent
preserved) not DELETE (contract abandoned).

This is a v2.11 follow-up candidate: the umbrella `_EXCEPTION_STATUS_MAP`
declares 409 for `ForbiddenTransitionError`, but the ticket-route
override silently overrides to 422. The map entry is now
dead code for this exception. Either remove the entry (acknowledging
the override is canonical) or normalise the envelope so a single
handler covers all `AppError` subclasses uniformly.

## G6 — Lessons (preview, full version in `ticketing-v2.10.md`)

1. **The SPA catch-all silently swallows everything.** WP01 noted it
   ate 404 assertions; WP07 found it also eats *test-only* routes
   registered after `create_app()` returns. The fix is to insert at
   the front of the route table or build a minimal `FastAPI()` that
   only attaches the exception handlers. A static-AST lint that flags
   `@app.get(...)` after `create_app()` would prevent the next
   instance.

2. **`patch("app.X.symbol", ...)` rots silently across refactors.**
   WP03 saw this with `app.auth.magic_link.get_settings`; WP07 sees
   it with `app.main._check_database`. v2.11 lint candidate: any
   `patch(str, ...)` where the dotted path doesn't resolve at
   collection time should fail collection, not at runtime.

3. **Schema drift is a contract drift.** `CommentResponse` grew four
   required fields, but the v1 schema tests never knew. Test-failure
   on `Field required` is the signal that the schema's *consumers*
   in production were updated but the *pin* on shape was missed.
   Worth a v2.11 schema-test sweep: every Pydantic model with a
   `model_rebuild()` call deserves a constructor-shape test that
   gets re-generated when the model changes.

4. **0% prod-bug yield in WP07 is not a problem — it's the design.**
   The v2.10 cluster's running prod-bug yield is now
   **2 / 313 = 0.64%**: one from WP02 (`agent_accounts.created_by`
   NOT NULL), one from WP04b (`search.py p.status` raw-SQL stale
   column). WP07 was framed as "decide PORT/REPLACE/DELETE per ID";
   the high fraction of PORT verdicts is the evidence that the v1
   surface is still load-bearing and the tests were salvageable.
