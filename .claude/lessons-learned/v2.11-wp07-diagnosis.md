## Frontend-shape risk findings

1. **`TicketRead` schema gap** — pre-WP07, the schema was missing eight
   fields that `Ticket.to_dict()` and the frontend `TicketDTO` reference.
   When wiring `response_model=TicketRead` on the new ticket endpoints,
   the missing fields would have been **silently dropped from every
   response**, breaking Kanban, TicketDetail, and CreateTicket flows.
   **Fix**: extended `TicketRead` to mirror `to_dict()` exactly, plus
   `extra="allow"` belt-and-braces. This also retroactively fixes the
   WP06 `search_tickets` route (which had been narrowing its response
   shape since WP06 shipped — caught here, not in WP06's tests).
2. **Leaderboard track-specific score keys** — the solvers and reporters
   responses return different score keys (`accepted_count` vs
   `upstar_count`). The frontend reads any of them via a fallback
   chain. Solved with `extra="allow"` on `LeaderboardEntry` so neither
   key is dropped.
3. **Claim toggle dual shape** — `claim_problem_route` returns either
   `{detail, claimed: false}` (claim removed) or `{detail, claimed: true,
   claim_id}` (claim added). `claim_id` is declared `str | None = None`
   on `ClaimToggleResponse`, so both shapes serialize cleanly.

## v2.11 follow-ups

- **B2-followup-1**: `app/routes/tickets.py::list_tickets` (the GET-collection
  endpoint) still returns ad-hoc `{items, next_cursor, total, column_counts}`.
  Doesn't fit plain `Page[T]` (the `column_counts` extra). Either define a
  `TicketsKanbanPage(Page[TicketRead])` subclass with the extra field, or
  split into a separate WP06-style ticket. Deferred.
- **B2-followup-2**: `list_comments` returns `{items: [...]}` without
  cursor/total. Eventually promote to `Page[TicketCommentRead]` once the
  frontend tolerates the envelope change (the WP06 lesson applies — verify
  no consumer reads ad-hoc keys before changing the shape).
- **B2-followup-3**: `_single_comment_response` (comments.py:258) is a
  free-floating dict helper consumed by `edit_comment_route`. It works
  today because the route's `response_model=CommentResponse` does the
  coercion, but it's fragile (the helper returns `replies: []` literal,
  while `CommentResponse.replies` is `list[CommentResponse]`). A small
  refactor to have the helper return a `CommentResponse` directly would
  remove the implicit coercion dependency.
