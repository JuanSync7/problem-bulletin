## Mypy config choices

```toml
[tool.mypy]
python_version = "3.12"
follow_imports = "silent"
warn_unused_ignores = true
warn_redundant_casts = true
exclude = ["alembic/", "tests/", "scripts/"]

[[tool.mypy.overrides]]
module = ["jose", "jose.*", "authlib", "authlib.*"]
ignore_missing_imports = true
```

### Choices justified

1. **`python_version = "3.12"`** — matches the `.venv` runtime. The
   repo's `requires-python = ">=3.11"`, but mypy's analysis target
   needs a concrete pin.

2. **`follow_imports = "silent"`** — `app/` imports from `tests/`
   helper modules in a few places; silent keeps the noise out of the
   per-file offender list.

3. **`warn_unused_ignores = true`** — keeps the v2.17-WP03 inline
   `# type: ignore` allow-list honest. Adding mypy revealed 2
   already-stale ignores that v2.17-WP03 could not have caught (it
   inspects pragmas, not their necessity); fixing both was the
   primary opportunistic-fix path.

4. **`warn_redundant_casts = true`** — same idea, applied to
   `typing.cast(...)`. Validates v2.18-WP04's cast as non-redundant.

5. **`exclude = ["alembic/", "tests/", "scripts/"]`** — `app/` is
   the production surface; tests / migrations / scripts stay opt-in
   for a future WP.

6. **Per-module `ignore_missing_imports = true` for `jose` /
   `authlib`** — neither has published type stubs (jose suggests
   `types-python-jose`, authlib has none). Suppressing per-module
   keeps the noise localised; we don't litter inline
   `# type: ignore[import-untyped]` pragmas across `app/auth/`.

### Choices deliberately NOT made

- **`pydantic.mypy` plugin** — the installed Pydantic version's
  plugin errors against mypy 1.20.1:
  `module 'mypy.expandtype' has no attribute 'ExpandTypeVisitor'`.
  This is a known version-mismatch issue. Until both packages catch
  up, the Pydantic-v2 BaseSettings env-driven constructor (8 hits at
  `app/config.py:172`) stays pinned BY-DESIGN with a single
  rationale. Revisit when mypy or pydantic releases a compatibility
  bump.

- **`sqlalchemy.ext.mypy` plugin** — would resolve the dominant
  `Column[T]` / `Mapped[T]` cluster (~130 of 164 errors). NOT enabled
  here because:
  a) The plugin requires the legacy `Mapped[T] = mapped_column(...)`
     declarative form; we are largely on `Mapped[T]`-typed columns
     but mixed with older `Column[T] = Column(...)` forms.
  b) Enabling the plugin would invalidate the entire PIN dict in
     one go and force a follow-up sweep — out of scope for WP02.
  Flagged as a v2.20+ candidate.

- **Strict mode (`strict = true`)** — we PIN-and-fix-forward, not
  bulk-tighten. A future WP can flip individual strictness flags
  (e.g. `disallow_untyped_defs`, `warn_return_any`) and re-pin.

## Lint shim implementation notes

The shim invokes `python -m mypy app` via subprocess and parses
`(path, line, errcode)` from each error line via regex:
`^(?P<path>app/[^:]+):(?P<line>\d+):\s*error:\s.*\[(?P<code>[a-z][a-z0-9_-]*)\]\s*$`.

`note:` lines are deliberately skipped — they're context, not
errors. The 8 `call-arg`s at `app/config.py:172` all collapse to one
allow-list key (same `path:line:errcode`); that's correct behaviour
and matches the v2.17-WP03 multi-pragma-per-line collapse.

The end-to-end synthetic-bad self-test (mypy must flag an obviously
broken file) is gated behind `RUN_MYPY_SELFTEST=1` to avoid
polluting the production `.mypy_cache/` on every test run. Five
parser self-tests prove the regex itself: single error, multi-error
multi-code, notes-and-summary skipping, hyphenated-code handling,
clean-output empty return.

## Runtime

- **Cold** (no `.mypy_cache/`): ~11.75s on `app/`. First-CI-run cost.
- **Warm**: ~0.5s -- well under 1s. Acceptable as a routine gate.

The test is decorated with `@pytest.mark.slow` so a future CI tier
can gate it via `-m "not slow"`. The mark is currently unregistered
(produces a `PytestUnknownMarkWarning`), but the test still runs;
registering the marker is a 1-line `pyproject.toml` addition a
future WP can do alongside other marker hygiene.
