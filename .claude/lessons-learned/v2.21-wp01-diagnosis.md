## E1 — `<Service>.list` shadow-of-builtin

### Root cause
A method named `list` on a service class shadows the builtin `list` *inside the class body*. Subsequent return-type annotations `list[Model]` on sibling methods are parsed by mypy as referring to the method, producing `[valid-type]` errors. Consumers downstream then see `list?[Model]` (mypy's fallback) and fail `[attr-defined]` on `__iter__`.

|---|---|---|
| `TicketService.list` | `app/services/tickets.py:628` | `async def list(self, session, *, status=…, …)` |
| `ProjectService.list` | `app/services/projects.py:127` | `async def list(self, session, *, include_archived=False) -> list[Project]` |

## E2 — `pg_insert(Model.__table__)` → `pg_insert(Model)` rewrite

### Root cause
`sqlalchemy.dialects.postgresql.insert` accepts a mapped class or a `TableClause`, but `Model.__table__` is typed as `FromClause` (the supertype), failing the overload. Passing `Model` directly resolves via the mapper overload.

## WP ordering recommendation

**WP03 = E1 (`<Service>.list` rename, 15 keys, ~19 sites)** second.
**WP04 = closure** (retrospective + v2.22 seed).

### Justification
Forward rule (mm) — never combine work-packages — applies. Both changes are code (not config), so size/risk drives ordering. E2 is smaller (8 mechanical 1-token edits, single grep-narrow file set, no test-caller rewrites needed because tests go through services, not `pg_insert` directly), so it lands first as the lower-risk warm-up. E1 is larger (19 callsite renames across both app and tests) and benefits from landing on a cleaner baseline where E2's resolved errors don't muddy the diff review.

Smaller-safer-first is the v2.13-style ordering that has worked reliably across the last 8 versions.
