# v2.13-WP04 (B3) — request-side contract pins for `*Create`/`*Update` schemas

## Scope

Mirror of v2.12-WP03. WP03 pinned the **response** side: every key
emitted by `ORM.to_dict()` must be declared on the matching `*Read`
schema (or the schema opts into `extra="allow"`). The inverse bug
class — backend accepts a `*Create` / `*Update` payload, Pydantic
validates it, but a field is silently dropped before reaching the
row — was unpinned. This WP closes it.

## Invariant

For each `(*Create | *Update schema, consumer)` pair: every name in
`schema.model_fields` must appear in the consumer's source (route +
service, unioned where the route hands off to a service) as either

* an attribute access (`payload.fname`, `data.fname`, `body.fname`,
  `self.fname`),
* a string-literal constant (catches `mutable = {"name", ...}` guard
  sets used by the PATCH services), or
* a keyword-arg name in a function call (catches kwarg fan-out
  `svc.create(title=payload.title, ...)`).

Polarity: all schemas in this WP are **closed** (default — no
`extra="allow"`), so the assertion is

```
schema.model_fields − consumer_referenced_names ⊆ excluded_fields
```

`excluded_fields` carries OCC tokens (`version`) that the route
deliberately strips via `exclude={"version"}` before forwarding to
the service.

## Polarity per pair

All schemas are closed (no `extra="allow"`). Polarity is therefore
the standard `schema ⊆ consumer` direction. No flipped-polarity pairs.

## Drift findings

**Zero drift detected.** All 17 pairs pass green on first run.

The codebase already has strong contract discipline on the request
side because:

* PATCH routes universally do `model_dump(exclude_unset=True)` and
  the service holds an explicit `mutable` set string-literal allowlist
  — the set IS the contract surface, and forgetting to add a field
  trips a `ValidationError("not updatable via update()")` at runtime
  the moment a caller exercises it. The lint formalises what was
  already enforced informally.
* POST routes universally use kwarg fan-out — adding a field to the
  schema without wiring it forces a manual edit at the route handler
  (the field would otherwise be unreferenced and obvious in review).
* `data.*` ORM-constructor patterns in the legacy `problems` /
  `solutions` / `comments` services pass every schema field straight
  into the model constructor.

This zero-drift result is the symmetric inverse of WP03 (which also
landed with zero drift on the first run after the WP07 narrowing was
fixed). The pin is a regression net, not an active cleanup.

## Deferred / out of scope

* Schemas with non-table consumers (e.g. query-builder bodies,
  realtime WS payloads, OAuth callbacks) are out of scope — this WP
  targets persistence-side drift only.
* `*Body` / one-shot ad-hoc schemas (`TicketTransitionBody`,
  `TicketAssignBody`, `TicketCommentBody`, `TicketLinkBody`,
  `TicketWatcherBody`, `TicketAttachmentBody`, `MagicLinkRequest`) —
  these aren't `Create`/`Update` per the seed grep, and would expand
  scope. Deferrable to a follow-up if needed.
