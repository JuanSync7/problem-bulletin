# v2.11-WP02 — Raw-SQL `legacy_status` sweep

## Context

Migration `a1_agent_kanban` renamed `problems.status` → `problems.legacy_status`.
The ORM mapping in `app/models/problem.py` keeps the Python attribute
`Problem.status` pointing at the renamed column (asymmetric on purpose — Bucket
E2 defers the full ORM rename). That asymmetry is invisible to ORM call sites
but fatal to raw-SQL fragments referencing `p.status` / `problems.status`.

v2.10-WP04b fixed one such hit in `app/services/search.py` (the
`test_search_filter_status` regression). This WP audits the rest of the repo
and pins the regression so the category can't reopen.

## Validatable end goals (declared before edits)

- **G1**: AST-based audit test scans `app/**/*.py` (excluding alembic) for
  raw-SQL `p.status` / `problems.status` literals. Deterministic listing.
- **G2**: Every hit either fixed to `legacy_status` or annotated.
- **G3**: Live-DB integration test exercises the previously-broken raw-SQL
  path. (Already exists: `tests/services/test_search.py::test_search_filter_status`.)
- **G4**: G1 audit lives as a regular pytest test (collection-time independent),
  so future drift fails CI not production.
- **G5**: Backend suite ≥ 1192 passed + new tests, 0 failed.
- **G6**: ORM-mapped uses of `Problem.status` (Python attribute) left alone.

## Regression-lint mechanism

**Chosen:** plain `pytest` test (not a `pytest_collection_modifyitems` /
`conftest` hook).

**Why:**

1. Symmetric with the v2.11-WP01 pattern (`__table__` introspection done via
   a normal test, not a conftest hook).
2. A collection-time hook would either skip silently when the audit module
   couldn't import or risk hiding the failure as a collection error rather
   than a clear, attributable test failure.
3. A regular test gives the failing literal's path + line number + snippet
   in the assertion message, which a collection hook can't surface as
   cleanly.
4. `tests/conftest.py` v2.10-WP07 explicitly removed the
   `pytest_collection_modifyitems` hook for v1-deferred handling; reintroducing
   a different collection hook here would re-muddy that boundary.

## Scanner design notes

- Uses `ast.walk` over `Constant(str)` nodes plus `JoinedStr` (f-string)
  literal segments — covers `text("...")`, plain strings, and f-string
  fragments. Misses templating only when a column reference is dynamically
  assembled from non-literal pieces, which would already be a code-review
  red flag.
- Regex: `r"\b(?:p|problems)\.(?!legacy_)status\b"`. Negative lookahead on
  `legacy_` keeps the correct spelling clean. Anchored on `p.` / `problems.`
  so unrelated entities (`t.status`, `tickets.status`, `flags.status`) are
  not flagged.
- Self-test (`test_audit_lint_detects_synthetic_drift`) writes three
  synthetic files into `tmp_path` and asserts the scanner flags the broken
  one, passes the correct one, and ignores the cross-entity one. This
  guards against a future refactor that silently neuters the lint.
- Docstrings are intentionally scanned: a docstring with an example
  `p.status` fragment is drift waiting to happen (next maintainer
  copy-pastes it into real SQL). Per the audit table, the three current
  docstring hits in `tests/services/test_search.py` describe the
  `p.status` -> `p.legacy_status` *history*; the scanner only runs against
  `app/`, so those test-side comments are out of its scan path.

## Caveats logged for follow-up

- **Bucket E2 (deferred):** rename `legacy_status` back to `status` and

## Lessons

- "Drift sweep" WPs sometimes find the production surface already clean —
  the value-add then becomes the **regression lint**, not a code fix.
  Documenting the empty-fix result is just as load-bearing as documenting
  the patches.
- A self-test on the audit scanner (RED on synthetic drift, GREEN on
  correct spelling, GREEN on unrelated entity) is cheap insurance against
  a future refactor accidentally turning the lint into a no-op.
- The negative-lookahead trick `\b(?:p|problems)\.(?!legacy_)status\b`
  is the cleanest way to express "the bad spelling but not the good one"
  without enumerating contexts. Worth reusing for future post-rename
  sweeps (e.g. when other columns get renamed via the same pattern).
