## Parity lint chosen — WP11 (inner-pair), not WP05 (wrapper)

The WP01 task brief said:
> If route returns bare `list[TicketWatcherRead]` → add to WP11_ROUTES.
> If route returns `Page[TicketWatcherRead]` → add to WP05_PAGE_PAIRS.

Route returns `Page[T]`, so the literal reading is WP05. However the
established precedent for `Page[T]`-returning routes is to pin the
**inner item shape** in WP11 (see existing WP11 entries for
`AgentActivityItem` and `TicketNotificationRead` — both routes use
`Page[T]` envelopes, but the WP11 entry pins the inner schema's
field set; the wrapper-level pin is owned by WP05_PAGE_PAIRS, which
keys on the generic `Page<T>` declaration in tickets.ts).

Adding a WP05_PAGE_PAIRS entry for `Page_TicketWatcherRead_` would be
**redundant** — the existing 6 PAGE_PAIRS entries all share the same
TS-side `Page<T>` declaration (tickets.ts:211); the lint reads that
single source every time. A 7th entry would re-assert the identical
`{items, next_cursor, total}` field set against the same TS source.

What WP11 adds is **new signal**: the 5-field `TicketWatcher` interface
has no prior contract pin. The WP11 inner-pair test exercises every
field name on both sides — that's the regression-deterrent value
this WP buys.

Parametrize delta: WP11_ROUTES **14 → 15** (+1). WP05_PAGE_PAIRS
unchanged at 6.

## Surprises

1. **Backend route ignores cursor/limit.** The route signature
   (`async def list_watchers(id_or_key, request, response, actor, db)`)
   has no Query parameters for pagination — it always returns the
   full watcher set wrapped in `Page[T]`. The TS helper signature
   reflects that (no params arg). If the backend ever adds
   cursor/limit, the TS side will need a `ListTicketWatchersParams`
   shim mirroring `ListNotificationsParams`.

2. **WP01's WP05-vs-WP11 framing was schema-symmetric but value-
   asymmetric.** Adding to WP05_PAGE_PAIRS would be the literal
   reading of the brief but would re-pin the same `Page<T>` field
   set against the same TS source as the existing 6 entries.
   Adding to WP11 pins a genuinely new field set. Chose WP11.

3. **Frontend tsc errors are pre-existing.** Running `npx tsc
   --noEmit` reports 76 pre-existing errors (mostly missing
   `global`/`@types/node` types in test files and unused `React`
   imports). My new code adds 0 new errors. The repo doesn't gate
   on tsc — vitest is the source of truth — so this matches v2.19's
   typecheck-as-CI PIN's actual baseline.
