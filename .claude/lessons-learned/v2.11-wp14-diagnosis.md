## Cursor payload schema — before/after

**Before (WP10):**

```json
{
  "rank": 0.5,
  "id": "00000000-0000-0000-0000-000000000001",
  "t": 117
}
```

**After (WP14, additive):**

```json
{
  "rank": 0.5,
  "id": "00000000-0000-0000-0000-000000000001",
  "t": 117,
  "a": "snapshot"
}
```

The new `"a"` field is one of `"snapshot"` or `"live"`. It is only
emitted alongside `"t"` (a cursor that doesn't carry a total
snapshot also has no authority statement to make).

## HMAC backwards-compat strategy

Per the brief, picked option (a): the helper accepts BOTH old-shape
(no `"a"`) and new-shape (with `"a"`) cursors during the transition.

The HMAC signature is computed over canonical-JSON of
`{"a": <arm>, "p": <payload>}`, where the canonicaliser uses
`sort_keys=True, separators=(",", ":")`. This means:

- A **WP10 cursor** has payload `{"rank", "id", "t"}` — the
  signature is bound to that key set. The HMAC verifier
  recomputes the signature from the *received* payload, so it
  still verifies cleanly — old keys, old sig, no problem.
- A **WP14 cursor** has payload `{"rank", "id", "t", "a"}` — the
  extra `"a"` enters canonical JSON and the new signature covers
  it. The verifier again recomputes from the *received* payload
  and accepts.

There is **no version field** on the envelope; none is needed
because the canonical-JSON sig naturally adapts to whichever key
set the payload happens to carry. The same shape-agnostic approach
is what made WP10 a non-breaking add-on for pre-WP10 cursors —
WP14 piggybacks on the same property.

Test that pins this: `test_legacy_cursor_without_authority_still_decodes`.

## Default-behaviour adjustments

None. `refresh_total` defaults to `False` on both the service
function and the FastAPI query param. With the default:

- The `"a"` field added to the next-page cursor is `"snapshot"`
  for every page (first page mints the snapshot; subsequent pages
  preserve `prior_authority` read from the incoming cursor, which
  for a clean WP14 chain is always `"snapshot"`).
- The returned `total_authority` follows the same rule.
- The WP10 stable-total invariant — "page 2 reports the same total
  as page 1 even if rows have been inserted/deleted" — is
  preserved unchanged. Pinned by
  `test_refresh_total_false_default_preserves_stable_total`.

When `refresh_total=True` is passed on any request (with OR
without a cursor):

- The arm's response `total` becomes the live `COUNT(*) OVER ()`
  computed in this request.
- `total_authority="live"` is surfaced.
- The next-page cursor minted from this response carries the same
  fresh total + `"a": "live"`. So if the client then drops
  `refresh_total` on the *next* page, the snapshot inherited from
  the cursor is the fresh value just established, with authority
  still labelled `"live"` (it was last re-counted, not snapshotted
  on page 1). This preserves auditability — the authority field
  tracks the provenance of the total, not just whether the
  current request opted in.

## v2.11 follow-ups

- **UI banner consuming `total_authority`** is intentionally
  deferred per the brief. Plumbing is in place: when product
  decides to surface "Showing snapshot count — refresh to update",
  the field is already on the response type and arm shape.
- **Refresh button on Search page** could be wired to call
  `searchV2({ ...currentParams, refresh_total: true })` without
  resetting the cursor. Trivial to add when the UI work lands.
- Consider whether `"a": "live"` should "reset" to `"snapshot"`
  on the next snapshot-mode request (i.e. when a request comes in
  with no cursor — the new snapshot is by definition fresh). The
  current code does that automatically: a cursor-less request
  always mints `"snapshot"`. Just calling it out so the contract
  is unambiguous if anyone audits the state machine later.
