# v2.10-WP11 — URL filter sync on Search

Status: in progress.

## G1 — Audit of current Search filters

Search.tsx currently keeps the following pieces of filter/scope state:

| State variable        | Type                  | URL today | URL after WP11 |
|-----------------------|-----------------------|-----------|----------------|
| `query`               | string                | `?q=`     | `?q=`          |
| `activeTab`           | SearchEntity          | `?entity=`| `?entity=`     |
| `problemStatus`       | string (ProblemStatus)| —         | `?problem_status=` |
| `problemCategoryId`   | uuid string           | —         | `?problem_category_id=` |
| `ticketStatus`        | TicketStatus or ""    | —         | `?ticket_status=` |
| `ticketProjectId`     | uuid string           | —         | `?ticket_project_id=` |
| `componentProjectId`  | uuid string           | —         | `?component_project_id=` |
| `page` (legacy/dead)  | number                | —         | not synced     |
| cursor stack (hook)   | array                 | —         | not synced (security) |

### Naming decision

Backend `/api/search/v2` already accepts these exact snake_case param names
(`problem_status`, `problem_category_id`, `ticket_status`, `ticket_project_id`,
`component_project_id`). Reuse them verbatim in the URL — zero translation,
easy to copy/paste a working API call from the URL bar, matches every
other route in the app (`?order_by=`, `?last_activity_at`).

`q` and `entity` already exist as-is; do not rename for backwards compat
with any bookmark made between WP57 and WP11.

## G2 — Sync semantics

Bidirectional sync rules:

- **URL → state (read)** is a one-shot seed at mount. We do **not**
  re-read URL params on every change beyond letting React Router's
  internal `useSearchParams` state drive the input values during
  back/forward navigation. The `useSearchParams()` hook already returns
  a live value, so reading off `searchParams.get(...)` on each render is
  free.
- **State → URL (write)** happens inside a `useEffect` that depends on
  every synced filter, using `setSearchParams(params, { replace: true })`
  so a five-keystroke debounced query does not pile up five history
  entries. Browser back from the Search page goes back to the previous
  *page*, not to the previous filter inside Search. This matches Gmail,
  Linear, Notion, GitHub issues.
- **Empty filters are omitted.** `?q=&entity=` is ugly, signals to bots
  that empty-string is a valid value, and confuses humans. We build
  the params dict and only set keys whose values are non-empty.
- **Cursors are NOT synced.** Rationale:
  1. Cursors are HMAC-signed and contain a server-side payload that
     leaks into the Referer header on outbound link clicks. Even though
     they're not secret per se, treating them as bookmarkable normalises
     handing them out.
  2. Cursor validity is tied to the current query + filter snapshot;
     bookmarking `?q=foo&cursor=<sig>` and revisiting a week later may
     fail HMAC verification or, worse, decode against new data.
  3. Browser back should restore filters, not deep-paginate position.
  4. The cursor stack is session-scoped state that the
     `useSearchV2` hook resets on every filter change anyway.

  This decision is identical to how GitHub does it: filter URL is shared,
  pagination cursor is not.
- **Single source of truth.** The URL is authoritative for filters; React
  state mirrors it. To avoid the classic "two sources race" we keep
  state local but reset state from URL only at mount. After mount, all
  filter mutations call `setSearchParams` and then update local state
  in the same handler. The write-effect dedupes via `replace: true`.

### Special case: query input + debounce

The `q` input is the only piece of state that has a debouncer in
front of it. Two natural options:

1. Push raw `query` to URL immediately, debounce only the network call.
2. Push debounced `debouncedQuery` to URL, network call uses the
   same debounced value.

Option 2 wins:

- URL doesn't churn per keystroke (`?q=h`, `?q=he`, `?q=hel`, ...).
- A back-button after typing returns the user to the previous *settled*
  state, not the previous keystroke.
- Matches existing WP57/WP58 behaviour for `q` (which was already
  syncing the debounced value).

The local controlled input `query` remains the live keystroke value;
`debouncedQuery` is what hits both the URL and the network.

### Invalid params

Unknown `entity` values (e.g. `?entity=pizza`) must not crash. The
component already does `(searchParams.get("entity") as SearchEntity)
?? "all"` — that defeats TypeScript but accepts garbage. Add a
runtime guard via a `TAB_IDS` set: if the URL value isn't a known
entity, fall back to `"all"`. Same posture for `problem_status` and
`ticket_status` — validate against the allowed enum, drop if invalid.
Project/category IDs we cannot validate without an extra round-trip,
so we accept any string; the backend rejects invalid UUIDs cleanly.

## G3 — Implementation plan

- Add a small `parseEntity(value: string | null): SearchEntity`
  helper at module top.
- Replace the existing on-mount seed for `q` / `entity` with a seed
  for all seven filters; gate `problem_status` and `ticket_status`
  against their enum sets.
- Extend the existing URL-sync effect to write every non-empty filter.
- Continue using `replace: true`.
- No new hook extraction — logic is contained enough to keep inside
  the page component, in keeping with the existing WP57 layout. Will
  revisit in v2.11 if a second surface (e.g. global typeahead) needs
  the same URL plumbing.

## G4 — Edge cases covered

- URL `?q=hello&entity=tickets&ticket_status=todo` on mount →
  state seeds correctly, single fetch with `ticket_status=todo`.
- Changing the ticket-status dropdown writes `ticket_status` into the
  URL; clearing it removes the key.
- Switching from Tickets tab to Problems tab does **not** clear the
  carried-over `ticket_status` URL key in the same render (state
  resets are tied to deliberate filter clears, not tab switches —
  matches existing behaviour where ticket-status state survives a
  tab switch). The write-effect, however, only writes filter keys
  that are non-empty *in state*, so a stale URL key from a previous
  visit gets removed on the first user interaction. This is a minor
  trade-off; alternative (clearing ALL filter URL keys on tab change)
  is more aggressive than what users typically expect.
- `?entity=pizza` → falls back to `all`, no crash.
- `?problem_status=banana` → ignored, state stays `""`.
- Browser back across two filter states inside Search keeps the
  filter inputs in sync with the URL because React Router's
  `useSearchParams` re-renders the component with new values when
  the history popstate fires; the on-mount seed code path runs
  again only if the component remounts — for in-page back/forward
  we need the URL→state direction to also flow on every render,
  not just mount.

That last bullet means the seed has to be re-evaluated whenever
`searchParams` changes — i.e. URL is the upstream value, state is
derived. The implementation handles this by reading `searchParams.get`
inline on each render and using it directly for inputs that aren't
keystroke-driven (entity tab, dropdowns), while the keystroke input
keeps a `useState` mirror seeded on mount + reset whenever the URL
`q` diverges from local state.
