Candidates that meet the WP11 lint shape (flat interface, named OpenAPI
schema, hand-written TS): **5**. Threshold ≥ 5 → proceed to expansion.

## Decision

**Expansion path**. The five candidates above are all flat interfaces
the WP11 parser handles, and every backend schema is already emitted
into `components.schemas` by transitive ``Page[T]`` references.

## Expansion outcome

Added 5 entries to `WP11_ROUTES` in `tests/test_openapi_ts_parity_wp11.py`:

1. `GET /api/v1/agents/activity` — `AgentActivityItem` ↔ `ActivityEntry` (audit.ts)
2. `GET /api/v1/audit-log` — `AuditLogEntryRead` ↔ `AuditLogEntry` (auditLog.ts)
3. `GET /api/v1/audit-log` — `PersonRef` ↔ `AuditLogActor` (auditLog.ts)
4. `GET /api/v1/people/search` — `PersonRef` ↔ `PersonRef` (people.ts)
5. `GET /api/v1/notifications` — `PersonRef` ↔ `PersonRef` (notifications.ts)

All five passed on first run — **zero drifts caught, zero fixes needed**.
The hand-written TS interfaces and the Pydantic schemas were already in
field-set agreement.

`PersonRef` is pinned three times (under three different routes / three
different TS-side type names: `AuditLogActor`, `people.PersonRef`,
`notifications.PersonRef`). Each entry exercises both:
- the route-existence assertion (catches the path moving), and
- the schema-vs-TS field-set comparison (catches drift on either side).

The duplication is intentional — the WP11 lint's `(method, path,
schema, ts_file, ts_type)` tuple is the addressable unit, and a single
backend schema can power multiple frontend contracts.

## Drifts caught + fixed

None. Five new entries, five clean pins. This is a meaningful signal:
the hand-written TS interfaces for activity feed, audit log, and the
PersonRef-reuse surfaces have been kept in sync through every WP that
touched them (v2.5-WP33 audit-log read API, v2.6-WP45 activity DTO,
v2.5-WP32 PersonPicker, v2.2-WP14 notifications inbox). No latent
drift.

## Deferred items (v2.23 backlog)

1. `SubtreeRow` ↔ `TicketSubtreeRow`: backend schema embeds `TicketRead`
   (`extra="allow"`) under `ticket:`. The WP11 parser doesn't descend
   into nested object literals on the TS side either, so a pin here
   would be vacuous. Re-evaluate if/when WP11 grows a recursive
   parser.
2. `AuditLogPage` / per-arm `Page<T>` shapes: lives in the WP05 lint
   surface, not WP11. WP05 already covers `Page[T]` generically; an
   explicit `AuditLogPage` pin would be redundant.
3. `TicketWatcherRead` / `TicketAttachmentRead`: frontend has no
   hand-written interface (consumers read items via `Page<unknown>` /
   inline shapes). Pin would require frontend writing the interface
   first — that's a v2.18-style any-tightening WP, not a parity-lint
   expansion.

## Methodology note

The WP11 lint's primary catch-vector is route-existence + schema-name
resolution + field-set agreement. With zero drifts caught on five new
entries, the marginal value of the expansion is **forward-looking**:
any subsequent change to `AgentActivityItem`, `AuditLogEntryRead`,
`PersonRef`, or the three TS-side consumers will now fail CI rather
than waiting for a runtime bug. This matches the v2.11-WP07 motivation
that originally seeded WP11.
